# Decision — Phase 2.5 ships as 4 sub-plans (2.5A → 2.5D)

**Date:** 2026-04-18
**Phase:** 2.5 (pre-execution)
**Spec:** [[superpowers/specs/2026-04-17-phase-2.5-biomarker-foundation-design|Phase 2.5 design spec]]

## Decision

Phase 2.5 ships as **four ordered sub-plans**, not one monolithic plan. Each sub-plan produces working, testable software on its own with its own first-pass code review before the next begins. Only Plan 2.5D's completion triggers the Phase 2.5 approval gate + merge to master.

## Why

- **Reviewability.** A monolithic 3000-line plan is hard to review and harder to course-correct. 4 plans of ~500–800 lines each map to discrete review cycles.
- **Deferred commitment.** Writing all 4 plans upfront commits us to detailed execution before we learn from executing earlier plans. Writing 2.5B after 2.5A merges gives us a chance to update based on what we found.
- **Precedent.** Phase 2 shipped as Plans 2A → 2B → 2C → 2D. This repo's workflow already fits the pattern.
- **Independent testability.** Each plan ends with something that works end-to-end in its own layer — you can invoke 2.5B's `parseLabReport` against a fixture PDF from the Convex dashboard long before any mobile UI exists.

## How to apply

- Plans are written lazily: 2.5B starts drafting only after 2.5A merges and the executing-agent feedback has landed.
- Each plan ends with its own first-pass code review committed to `docs/superpowers/reviews/YYYY-MM-DD-phase-2.5<letter>-<short-name>-review.md`. 2.5A's review is NOT the approval gate; it's a per-plan quality check.
- The **Phase 2.5 approval-gate checklist** (spec §9.5) runs only at the end of 2.5D. Nothing ships to master until then.
- If scope discovered during 2.5B/C/D shifts the architecture, we amend the spec + write a new decision record rather than silently drifting.

## Risks

- **Schema evolution across plans.** 2.5B/C/D depend on the schema shape locked in 2.5A. If a plan discovers a needed field, it must be a widen-migrate-narrow change documented in that plan's as-built retro, not a silent patch to 2.5A's schema commit.
- **Long-lived worktree.** Each plan runs in its own worktree (`D:/onlyou2-phase-2.5<letter>`). Merging 2.5A to master before 2.5B begins keeps master in a deployable state at every sub-plan boundary.
