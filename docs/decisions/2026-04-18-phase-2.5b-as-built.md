## Notable deviations from plan (decided during implementation)

- **`cache_control` on system block, not few-shot block** — the plan's architecture section said to cache "system + 8 few-shot examples." Implementation placed the `cache_control` breakpoint at the end of the system message only, because the few-shot block is per-call content that varies by fixture format. Documented in `docs/decisions/2026-04-18-cache-breakpoint-on-system-block.md` (`3bbfe40` on master).
- **Telemetry hash is pure-JS FNV-1a** — `convex/lib/telemetry.ts` uses a pure-JS FNV-1a hash instead of Node `crypto.createHash("sha256")`. The Convex V8 bundler does not expose Node built-ins in the default runtime; `"use node"` was not needed anywhere in this phase (no file-system or streaming operations). Documented in `docs/decisions/2026-04-18-telemetry-hash-pure-js.md` (`d088a4b` on master).
- **`FEW_SHOT_EXAMPLES` deferred to 2.5C** — the plan called for 8 few-shot extraction examples (one per fixture format) to be embedded in the system prompt. During review (I-3), this was deferred: populating the examples correctly requires the live API to validate round-trip fidelity, which belongs with the 2.5C ingestion phase when the prompt is tuned against real uploads. Logged in `docs/DEFERRED.md`.
- **`unit_conversions` field mapping uses `fromUnit`/`toUnit`/`factor`** — the plan described the JSON shape abstractly; implementation settled on a flat `{ fromUnit, toUnit, factor }` row format matching the existing `unit-conversions.json` seed, rather than a nested object.
- **`"use node"` directive not needed** — the plan anticipated that `parseLabReport` might need `"use node"` for PDF buffer handling or Node crypto. In practice, Convex file storage returns a `Blob` that the Anthropic SDK accepts as `base64` without Node streams, and telemetry hashing used pure JS. No `"use node"` was needed.
- **Public action path uses `api.*` not `internal.*`** — the plan's file structure placed `parseLabReport.ts` under `convex/biomarker/` with the expectation it would be an internal action triggered by `intakeUpload`. For dashboard testing in the 2.5B scope (before `intakeUpload` exists), it is registered as a public action callable via `api.biomarker.parseLabReport`. It will be converted to internal in 2.5C when `intakeUpload` wraps it.
- **Atomic cron lock fixed at review (I-1)** — the initial cron implementation had a TOCTOU race: it read the lock, checked it, then wrote — two cron firings could both claim the same report. Review finding I-1 introduced an atomic `lockedAt` swap inside a single mutation transaction, preventing double-fire.
- **400 HTTP status as terminal failure (C-1)** — the initial retry scheduler treated all non-429 HTTP errors as retryable. Review finding C-1 made 400-range non-429 errors terminal (flip to `failed`, alert tier `p1`), preventing infinite retry loops on structurally invalid payloads.
