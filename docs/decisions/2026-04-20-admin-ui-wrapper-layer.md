# Decision — Single wrapper layer at `apps/admin/src/components/ui/*` is the only shadcn import site

**Date:** 2026-04-20
**Phase:** 2.5C brainstorm
**Status:** Approved by founder during Plan 2.5C brainstorm.
**Related:** [[2026-04-20-admin-component-ceiling]], [[2026-04-20-admin-token-register]]

## Decision

Every shadcn component is imported, themed with `operatorPalette` tokens, and re-exported exactly once from `apps/admin/src/components/ui/*`. The rest of the admin app (pages, features, flows) imports shadcn components only via those wrappers — never from `@/components/ui` files generated by shadcn CLI directly, never from `@radix-ui/*` directly.

ESLint rule enforces the invariant:

```
no-restricted-imports:
  - patterns:
    - group: ["@radix-ui/*", "cmdk"]
      message: "Import themed wrappers from @/components/ui, not Radix directly."
    - group: ["./components/ui/raw/*"]
      message: "Use themed wrappers at @/components/ui/*, not raw shadcn files."
```

(Where `components/ui/raw/*` is a suggested sub-folder for the unthemed shadcn-CLI output — kept separate from the themed wrapper files.)

## Alternatives considered

1. **No wrapper layer — shadcn imported directly with Tailwind class overrides per usage site.** Rejected. Theme tweaks require touching every call site. Invariant "all admin Dialogs look like THIS" becomes a cultural rule that erodes the first time someone skips it.

2. **Global CSS variable override at the app root.** Rejected. shadcn reads tokens from CSS vars, so in theory you can set `--background` etc. at the root and shadcn respects it. Works for palette-level theming. Doesn't work for structural overrides (different padding, different border-radius, different focus ring). A wrapper layer absorbs both kinds of override cleanly.

3. **One wrapper layer AND direct shadcn imports for "simple" components.** Rejected. The line between "simple" and "needs theming" erodes the first time someone decides Skeleton is simple. Single rule is enforceable; graduated rules aren't.

## Why

- **Token substitution in one place.** When the design system shifts (Phase 5 operator-desk expansion, or a future palette tweak), the change is N wrappers — not N × call-sites.
- **A11y improvements propagate.** If Radix ships a bugfix in Dialog's focus-trap logic, bumping shadcn surfaces it in one wrapper. If shadcn's wrapper pattern improves, we absorb improvements in one file.
- **Visual audit is a directory listing.** The wrapper layer IS the audit — every themed component lives there. Reviewers don't have to grep for shadcn imports to know what was themed.
- **ESLint guardrail makes it real.** Without the rule, the wrapper becomes a "convention" and drifts. With the rule, violations are visible in CI.

## How to apply

1. **Directory layout:**

   ```
   apps/admin/src/components/ui/
     raw/           — unthemed shadcn output (generated by `shadcn add`)
       button.tsx
       dialog.tsx
       ...
     button.tsx     — themed wrapper, exports ThemedButton as default Button API
     dialog.tsx     — themed wrapper
     ...
   ```

2. **Wrapper pattern** — each themed file imports from `./raw/<name>`, composes with operator tokens, re-exports with the same API:

   ```ts
   import { Button as RawButton } from "./raw/button";
   import { operatorPalette } from "@onlyou/core/tokens/operator";
   // ...theme the component...
   export { Button };
   ```

3. **Imports elsewhere** always take the themed form:

   ```ts
   import { Button, Dialog } from "@/components/ui";
   ```

4. **ESLint rule landed** in `apps/admin/.eslintrc` during the 2.5C scaffolding task.

5. **Tests on the wrappers** verify the operator tokens come through — not shadcn defaults. One snapshot test per wrapper is enough.

## Known risks

- **Rule escape hatches.** An engineer disables the ESLint rule locally for a one-off. Mitigation: disabling `no-restricted-imports` in this repo is itself reviewable.
- **Wrapper API divergence from shadcn.** If the wrapper changes the API, upgrading shadcn becomes painful. Mitigation: wrappers keep identical exported API — they thin-layer; they don't re-invent.
- **Over-wrapping.** Wrapping a component we only use once, the same way, is ceremony. If a component stays unused-except-once for three months, it belongs elsewhere. Review at each phase-boundary.

## Reversibility

Removing the wrapper layer means a search-and-replace of imports + deleting wrappers. A few hours. Low cost, low value reversal — the wrapper layer is cheap to maintain, so "un-doing" it rarely pays off.
