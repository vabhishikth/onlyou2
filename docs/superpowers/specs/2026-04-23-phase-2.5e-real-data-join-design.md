# Phase 2.5E — Real-data join (reference ranges + trend + prev + personalization)

**Status:** Design approved 2026-04-23. Ready for planning.
**Prerequisite:** Phase 2.5D merged (`82e6f46`).
**Gate:** hard-gated before any external demo of the patient dashboard. Placeholder bounds (`low=0/high=100/optLow=25/optHigh=75`) visibly clip real markers on cholesterol 185 and LDL 165 — see Android parity screenshots at `docs/superpowers/phases/2.5d/wave-5-parity/`.

---

## 1. Goal

Close the hard gate that blocks external patient-dashboard demos by:

1. Joining real `biomarker_reference_ranges` bounds into the patient query and hook transform.
2. Joining multi-point historical trend + prior-report `prev` into each marker.
3. Promoting `action_required` status to direction-aware `"high"` / `"low"` so the range-bar rail renders correctly.
4. Wiring Dashboard greeting + date + avatar to the session user and live IST clock.
5. Replacing the hardcoded Detail byline with a brand-voice line (real doctor attribution deferred to Phase 4).
6. Making the Detail explainer copy status-aware so `LDL 165` no longer reads "within the optimal window" on an `action_required` marker.

This is the demo-ready shape. It does not require clinical sign-off on the 45 DRAFT reference-range rows — that is a parallel human-workflow track, tracked separately, still required before prod seed.

## 2. Scope

**In scope** (items 1–9 from `docs/DEFERRED.md` §"Phase 2.5E — hard gate before next demo", plus Wave 6 code-review perf folds):

1. Real reference-range join (`optimalMin` / `optimalMax` / `actionBelow` / `actionAbove`).
2. Multi-point trend via `by_user_canonical_date` index lookup per marker.
3. Prior-report `prev` derived from the second-most-recent point in the trend.
4. Direction-aware `action_required` → `"high"` / `"low"` mapping.
5. Dashboard greeting / date / avatar wired to session user + IST time-of-day.
6. Detail byline → brand voice (`"— Your care team"`).
7. Status-aware explainer copy (per-status variant per canonical id + fallback).
8. M-3 perf fold — soft-delete filter → index-backed where simple.
9. M-4 perf fold — `Promise.all` over canonical + trend lookups per marker.
10. Env-config verification doc — production `EXPO_PUBLIC_USE_MOCK_BIOMARKERS=0` is a merge-gate check.

**Out of scope** (explicitly deferred):

- Personalized ranges by patient sex/age beyond what the parse pipeline already stamps via `findReferenceRangeId`. The direct-pointer fallback on `val.referenceRangeId` is already sex/age-correct when parse matched; the sex-agnostic fallback runs only when no range matched at all, i.e. when there are no bounds to personalize anyway.
- Real doctor attribution for the Detail byline. No `doctor` field exists on `biomarker_reports` today. Added in Phase 4 (doctor portal).
- iOS parity screenshots — Mac-setup sidecar. Captured when hardware available. Lands in `docs/superpowers/phases/2.5d/wave-5-parity/` (not 2.5e; it is parity for 2.5D work).
- Pregnancy / post-menopausal reference-range data. PCOS vertical phase (range safety GUARD already shipped in 2.5).
- Clinical advisor sign-off on 45 DRAFT rows. Human workflow, parallel track, prod-seed prerequisite.

## 3. Architecture

### 3.1 Data flow

```
biomarker_reports (latest by analyzedAt desc)
  └── biomarker_values (per current report)
        ├── canonical join: referenceRangeId → biomarker_reference_ranges (preferred)
        │                   or by_canonical_id fallback (first active range)
        ├── range projection: { optimalMin, optimalMax, actionBelow, actionAbove }
        ├── trend lookup: biomarker_values.by_user_canonical_date
        │                 filter deletedAt === undefined, numericValue !== undefined
        │                 ascending by collectionDate, truncated at current report date
        └── prev = trend[trend.length - 2] ?? null
```

All per-marker work runs under a single `Promise.all` per report to fold the M-4 observation.

### 3.2 Query contract

File: `convex/biomarker/patient/myBiomarkerReports.ts`.

Projected `values[]` entries add three fields:

```ts
range: {
  optimalMin: number;
  optimalMax: number;
  actionBelow: number | null;
  actionAbove: number | null;
} | null;

trend: { value: number; collectionDate: number }[]; // ascending

prev: { value: number; collectionDate: number } | null;
```

Existing fields (`_id`, `canonicalId`, `nameOnReport`, `valueType`, `rawValue`, `rawUnit`, `numericValue`, `status`, `classifiedAt`, `canonical`) remain unchanged.

`range` is `null` when there is no canonical match (unclassified markers). `null` on individual `actionBelow` / `actionAbove` fields is preserved from the schema and handled in the hook.

Trend is returned server-side fully — the client slices for sparkline width. The query is patient-scale (one patient, a few dozen markers, a few reports per marker), so "return all and slice in the client" is correct here; revisit if growth changes that.

### 3.3 Auth + authz

Unchanged. Session-token pattern per `docs/decisions/2026-04-20-session-token-auth-in-2-5c-mutations.md`:

```ts
const session = await ctx.db.query("sessions").withIndex("by_token", …).unique();
if (!session || session.expiresAt < Date.now()) throw ConvexError("unauthenticated");
const caller = await ctx.db.get(session.userId);
if (!caller || caller.role !== "PATIENT") throw ConvexError("forbidden");
```

### 3.4 Perf folds

- **M-3** — the soft-delete filter currently uses `.filter(q.eq(q.field("deletedAt"), undefined))`. If an index-backed query using the existing `by_deleted` index simplifies it without materially changing semantics, use it. If the filter reads stay well under limits at patient scale, leave as-is and document. Measurement first; don't index-optimize on reflex.
- **M-4** — replace the serial `await ctx.db.get(val.referenceRangeId)` and serial canonical-fallback queries with a single `Promise.all` per report. The new trend query is folded into the same `Promise.all` so every marker's canonical + range + trend + prev data is fetched concurrently.

## 4. Hook transform

File: `apps/mobile/src/hooks/use-biomarker-reports.ts`.

### 4.1 Behavior change

Switch from flattening **all reports' values** into one list, to returning **only the latest report's values** with history encoded in each row's `trend` + `prev`. Dashboard shows "current state, with history"; Detail already filters by id so is unaffected.

### 4.2 Bound mapping + range direction

```ts
const optLow = r?.optimalMin ?? 25;
const optHigh = r?.optimalMax ?? 75;
const span = optHigh - optLow;
const low = r?.actionBelow ?? Math.max(0, optLow - span);
const high = r?.actionAbove ?? optHigh + span;

type RangeDirection = "bidirectional" | "unboundedLow" | "unboundedHigh";
const rangeDirection: RangeDirection =
  r?.actionBelow == null && r?.actionAbove != null
    ? "unboundedLow"
    : r?.actionBelow != null && r?.actionAbove == null
      ? "unboundedHigh"
      : "bidirectional";
```

Synthesizing bounds alone is not enough. `RangeBar` today renders a hardcoded red-amber-green-amber-red gradient (`viz/RangeBar.tsx:116-147`) — the red/amber stops are fixed percentages of the track, not computed from `low`/`high`. A synthesized `actionAbove` therefore still paints a red "danger zone" on the right edge for markers like HDL, where clinically there is no danger zone above optimal. That is a first-impression visual bug on exactly the demo this phase unblocks.

**Fix in scope:** add a `direction` prop to `RangeBar` (`"bidirectional"` default, `"unboundedLow"`, `"unboundedHigh"`). Gradient stops adjust so the unbounded side terminates in sage (green) rather than rose (red) — the honey (amber) stop also moves or drops on that side. Hook passes `rangeDirection` through `BiomarkerMock` to the card and detail consumers. Unit tests assert the stop list per direction.

This is a ~15-line `RangeBar.tsx` change, one new prop on `BiomarkerMock`, and one new field threaded through `BiomarkerCard` + `DetailHero`. Scoped as part of W2.

### 4.3 Direction-aware status

```ts
function mapStatusDirection(
  status: ConvexStatus,
  value: number,
  r: Range | null,
): BiomarkerStatus {
  if (status === "optimal") return "optimal";
  if (status === "sub_optimal" || status === "unclassified") return "watch";
  // action_required
  if (r && value < r.optimalMin) return "low";
  return "high"; // default when range missing or value above optimalMax
}
```

### 4.4 Trend handling

```ts
const trend =
  v.trend.length > 1 ? v.trend.map((p) => p.value) : [v.numericValue ?? 0];
```

Single-point trends still render (sparkline already handles `data.length === 1` → "now" label from 2.5D Wave 6 fix `918c9de`).

**Dropping `collectionDate` at the hook boundary is intentional.** Verified against current consumers:

- `viz/Sparkline.tsx:30-31` takes `data: number[]`, even X-spacing via `(w-4)/(data.length-1)`.
- `viz/AreaChart.tsx:50-51` takes `data: number[]`, X from `(i / (data.length - 1 || 1)) * iw`.
- `DetailHero` + Detail screen reference no per-point timestamps.

No current or near-term consumer needs per-point dates. Per-report timestamps live on `biomarker_reports.analyzedAt`, which is already projected. If a future timeline view needs per-point dates, add `trendPoints: { value, collectionDate }[]` alongside `trend: number[]` then — cheap, non-breaking.

### 4.5 prev

```ts
prev: v.prev?.value ?? (v.numericValue ?? 0),
```

Falls back to current value on first report, preserving today's zero-delta display rather than showing a spurious `–current`.

## 5. UI personalization

### 5.1 Dashboard (`apps/mobile/app/lab-results/index.tsx`)

- **Greeting** — `user.name` from `useAuthStore` (first space-delimited token). Fallback `"there"`.
- **Time-of-day prefix** — IST via `Intl.DateTimeFormat('en-IN', { timeZone: 'Asia/Kolkata', hour: 'numeric', hour12: false })`. `< 12` → `"morning"`, `< 17` → `"afternoon"`, else `"evening"`.
- **Date line** — IST via `Intl.DateTimeFormat('en-IN', { timeZone: 'Asia/Kolkata', weekday: 'long', day: 'numeric', month: 'long' })`. Uppercase. Reformat to `"WEEKDAY · DAY MONTH"`.
- **Avatar letter** — first character of `user.name` uppercased. Fallback `"?"`.

### 5.2 Detail (`apps/mobile/app/lab-results/[id].tsx`)

- **Byline** — replace `"— Clinical note, Dr. M. Rao"` with `"— Your care team"`. Inline comment: `// DEFERRED(phase-4): real doctor attribution when biomarker_reports.doctorId ships`.

### 5.3 Status-aware explainers (`apps/mobile/src/data/biomarker-explainers.ts`)

New type:

```ts
type Explainer = {
  base: string;
  byStatus: {
    optimal: string;
    watch: string;
    high: string;
    low: string;
  };
};
type ExplainerMap = Record<CanonicalId, Explainer>;
```

**Lookup key contract.** `byStatus` is keyed by the hook-mapped `BiomarkerStatus` (`"optimal" | "watch" | "high" | "low"`) from §4.3, **not** the raw Convex status. Reasoning: the Detail consumer receives `BiomarkerMock.status: BiomarkerStatus`, which has already been run through `mapStatusDirection()`. The raw Convex values (`"sub_optimal"`, `"action_required"`, `"unclassified"`) are gone by the time Detail renders. W4 implementation and the Detail consumer therefore must agree on the hook-mapped key — any future introduction of a new raw-Convex status requires updating `mapStatusDirection()` first, not the explainer map.

Detail screen composes `${base}\n\n${byStatus[status]}`.

For canonical ids with no explainer entry (unclassified, out-of-seed): fallback `"This marker is outside our reference database. Your care team will review it."`.

Copy drafted in the plan phase, founder visual-approved before merge.

## 10. Risks + open questions

- **Measurement for M-3.** We may leave the filter as-is if scans are small. Don't pre-optimize. Decision documented in the plan with actual numbers.
- **Status-aware copy coverage.** Founder review after drafting all 24 entries — if any read awkwardly in one of the four status states, the wave slips a copy iteration before merge. Acceptable.
- **Trend alignment across collectionDate vs analyzedAt.** `biomarker_reports.analyzedAt` is the parse timestamp; `biomarker_values.collectionDate` is the specimen collection date from the report. Trend uses `collectionDate`. Single-report history uses the same point. Spec calls this out; plan verifies no mismatch.
- **Reactive cost.** Full trend in every query invalidation re-reads all history. At patient scale this is cheap (low hundreds of reads). Revisit if a patient accumulates hundreds of reports — won't happen organically inside Phase 3.
