## 1. Goal + scope

Build the cross-vertical biomarker infrastructure once so every vertical afterward plugs in. Ship an end-to-end patient upload flow (pick PDF/JPG/PNG/HEIC → parse → classify → render a Visual Biomarker Report) plus the backend contracts for diagnostic-center upload (Phase 6 portal) and doctor-ordered labs (Phase 3+ nurse flow, Phase 4 doctor portal). **No phase after 2.5 rebuilds biomarker infrastructure.**

### 1.1 In scope (2.5)

**Backend**

- `biomarker_reference_ranges` table seeded with 32 markers (~45 reference-range rows) spanning all 5 verticals + CBC essentials + lipid-separated (HDL/LDL/Trigs).
- `biomarker_curation_queue` table auto-populated when the parser encounters unknown markers.
- `lab_reports`, `biomarker_reports`, `biomarker_values`, `lab_orders`, `parse_rate_limits`, `notifications` tables.
- `parseLabReport` Convex action — Claude vision-native extraction → schema validation → unit normalization → classification against ranges with pregnancy guard → narrative generation → notification emission.
- Curation backfill job (`reclassifyForCanonicalId`) — resolves queue items into reference-range rows and reclassifies affected `biomarker_values`.
- Dev-only admin stub for lab-upload path (`convex/admin.ts` + `scripts/seed-lab-upload.ts`) until Phase 6 lab portal ships.
- Rate limiter (5/day + 50/month per user) + kill-switch feature flag `biomarker_parsing_enabled`.
- `LAB_PORTAL_ENABLED` + `DOCTOR_PORTAL_ENABLED` env flags gating portal mutation registration (no live-but-dead endpoints).

**Mobile (patient app)**

- 4 screens ported from the design bundle: Dashboard, Biomarker Detail, Report, Upload.
- `apps/mobile/src/components/biomarker/` package: `Dial`, `RangeBar`, `Sparkline`, `AreaChart`, `MarkerCard`, `StatusBadge` (with 4th `unclassified` variant), `NarrativeCard`, upload-flow state components.
- Hybrid design-system refresh: Instrument Serif + JetBrains Mono fonts added alongside Playfair Display (Clinical Luxe); editorial light palette (`biomarkerPalette`) added to `@onlyou/core/tokens` as a second register.
- `pregnancyStatus` field added to profile; one-time inline bottom sheet on first upload when `gender === "female" && pregnancyStatus === null`.
- In-app notification banner + Activity-tab entry when any source produces a new biomarker report.

**Admin (web)**

- Curation admin surface at `/admin/biomarker/curation` + `/admin/biomarker/ranges`: minimum-viable tables + forms for triaging unknown markers and editing reference ranges.

### 1.2 Out of scope (explicit deferrals — see [[DEFERRED]] for full entries)

- Real push notifications → **Phase 8**.

### 2.10 `featureFlags` (existing, new key)

- `biomarker_parsing_enabled: true` — kill switch; when false, uploads queue with `status: uploaded`, no Claude calls fire, no rate-limit increments.

### 2.11 Cascade soft-delete

`deleteLabReport` mutation sets `deletedAt` on `lab_report` AND all owning `biomarker_report`s AND their `biomarker_value` rows in one transaction. All reads filter `deletedAt == null`. A single daily cron sweeps rows where `deletedAt < now - 30d`, deleting child-first (values → reports → lab_reports) then nulling `biomarkerReportId` + `labReportId` on referencing `notifications`.

## 3. Parse pipeline

### 3.1 Pipeline overview

```
Client upload → intakeUpload (mutation) → schedules parseLabReport (action)
parseLabReport steps:
 0. flag check → biomarker_parsing_enabled? if false, leave status=uploaded, return
 1. status → analyzing (optimistic lock on prior status)
 2. load bytes from storage
 3. compute contentHash server-side; dedupe check; if dup → reject
 4. PDF password check via pdf-lib headers → reject if encrypted
 5. Claude call #1: extraction (vision)
 6. zod schema validate → on fail, ONE retry with "fix the JSON" follow-up
 7. patient-name match (match | mismatch | unknown)
 8. per marker: normalize name → alias → canonicalId; normalize unit; classify
 9. for unclassified:not_in_reference_db → upsert biomarker_curation_queue
 10. insert biomarker_values rows + recompute aggregates
 11. Claude call #2: narrative generation
 12. insert biomarker_reports envelope
 13. lab_reports.status → ready; if source was lab_upload/nurse_flow, flip lab_orders.status → results_uploaded
 14. insert notifications row (lab_report_ready)
```

### 3.2 Client-side intake

- `expo-document-picker` for PDF; `expo-image-picker` for JPG/PNG/HEIC.
- **HEIC conversion** via `expo-image-manipulator.manipulateAsync(uri, [], { format: JPEG, compress: 0.9 })`; retry at `compress: 0.7` if still >20MB; reject with "This image is too large. Try a smaller photo or a PDF." if second pass also fails.
- Size + mime rejection client-side before upload. Upload via Convex `generateUploadUrl` + chunked PUT with progress.
- Client sends no hash; server computes authoritatively.

### 3.3 `intakeUpload` mutation

Arg shape (no `contentHash`):

```ts
args: {
  fileId: v.id("_storage"),
  mimeType: v.union(
    v.literal("application/pdf"),
    v.literal("image/jpeg"),
    v.literal("image/png"),
  ),
  fileSizeBytes: v.number(),
  source: v.union(
    v.literal("patient_upload"),
    v.literal("lab_upload"),
    v.literal("nurse_flow"),
  ),
  labOrderId: v.optional(v.id("lab_orders")),
}
```

Guards:

- Auth: `source: patient_upload` only for the session's userId. `lab_upload`/`nurse_flow` rejected here; they enter via `labPortal.ts` / `admin.ts`.
- `fileSizeBytes > 20MB` → reject with `file_too_large`.
- Rate-limit check: returns `rate_limited` with `which: "day" | "month"` + `resetAt` ISO timestamp. Client renders day or month copy based on `which`.
- Insert `lab_reports` with `status: "uploaded"`; increment `parse_rate_limits` (both windows); `scheduler.runAfter(0, parseLabReport, { labReportId })`.

### 3.4 Claude call #1 — extraction (vision)

Model centralized in `convex/lib/claude.ts`:

```ts
export const MODEL_EXTRACTION = "<verified-at-implementation>";
export const MODEL_NARRATIVE = "<verified-at-implementation>";
```

Both verified against current Anthropic docs at build time via Context7 / web fetch; never hard-coded from memory. No Opus escalation path in 2.5 (retry on same Sonnet model with "fix the JSON" follow-up; Opus carve-out deferred).

Request shape: system prompt (cached) + few-shot examples for 3 major Indian lab formats (Thyrocare, Metropolis, SRL) (cached separately) + user document content block (PDF base64 or image). Structured JSON output per schema:

```json
{
  "is_lab_report": boolean,
  "patient_name_on_report": string | null,
  "collection_date": string | null,    // ISO YYYY-MM-DD
  "markers": [
    {
      "name_on_report": string,
      "canonical_id_guess": string | null,
      "qualifier": string | null,
      "value_type": "numeric" | "qualitative",
      "raw_value": string,
      "numeric_value": number | null,
      "raw_unit": string | null,
      "lab_printed_range": string | null,
      "page_number": number,
      "confidence": number
    }
  ]
}
```

**Retry policy**: JSON validation fail → one retry with follow-up user message. Claude 5xx / rate-limit → exponential backoff (3 attempts over 2 min). Persistent fail → `parse_failed` + notification.

### 3.5 Classification (revised order — pregnancy before profile)

Per marker, first guard that fires wins:

1. **No canonical match** → `unclassified:not_in_reference_db` + curation queue upsert.
2. **Qualitative value** → `unclassified:qualitative_value`.
3. **Unit conversion missing** → `unclassified:unit_conversion_missing`.
4. **Pregnancy guard** — `user.gender === "female"` AND `user.pregnancyStatus ∈ {"pregnant", "unknown", null}` AND reference-range row has `pregnancySensitive: true` → `unclassified:pregnancy_sensitive`. Only `pregnancyStatus: "not_pregnant"` permits classification on sensitive markers.
5. **Profile incomplete** — missing `gender` or `dob` → `unclassified:profile_incomplete`.
6. **Classify against thresholds** (INCLUSIVE-INCLUSIVE):
   ```ts
   if (value >= optimalMin && value <= optimalMax) status = "optimal";
   else if (
     subOptimalBelowMin != null &&
     value >= subOptimalBelowMin &&
     value < optimalMin
   )
     status = "sub_optimal";
   else if (
     subOptimalAboveMax != null &&
     value > optimalMax &&
     value <= subOptimalAboveMax
   )
     status = "sub_optimal";
   else status = "action_required";
   ```

### 3.6 Unit normalization

`packages/core/seeds/unit-conversions.json` (~15 rows). Pattern-matched case-insensitive + punctuation-normalized. Covers mg/dL ↔ mmol/L, g/dL ↔ g/L, nmol/L ↔ ng/mL equivalents, mIU/L ↔ µIU/mL, nmol/L ↔ ng/dL (testosterone), pmol/L ↔ pg/mL (estradiol / free testo).

### 3.7 Patient-name match

Normalize: lowercase, strip punctuation, collapse whitespace, remove titles (`mr`, `mrs`, `ms`, `dr`, `shri`, `smt`).

Decision tree:

1. Report name absent → `unknown` (soft — allow).
2. Every ≥2-char token from profile name appears in report name (order-insensitive) → `match`.
3. Single-token profile vs multi-token report → require full-word match in report.
4. 1-char profile tokens (initials) → accept against any report token starting with that char in the same relative position.
5. Fallback: Jaro-Winkler on full normalized strings, threshold **0.88** → `match` ≥, else `mismatch`.
6. Both sides multi-token with partial overlap and a ≥4-char non-matching token → `mismatch`.

`mismatch` → `status: "rejected"`, `errorCode: "patient_name_mismatch"`, user-facing "If this is your report, contact support with a copy of your ID."

### 3.8 Claude call #2 — narrative

Text-only, no document. Input = classified marker summary (canonicalId, displayName, status, category — no numeric values). Prompt generates 2–3 sentences referencing only markers present, present tense, no medical advice, no specific numbers. `temperature: 0.3`. System prompt cached.

Narrative is **FROZEN** at initial analysis. Curation backfill does not regenerate. Post-foundation: regenerate on any `unclassified → classified` flip affecting ≥1 marker in the report.

### 3.9 Cost

Per parse with prompt caching on system + few-shot: ~$0.014 (~₹1.20). 10K parses/month ≈ ₹12K.

### 3.10 Idempotency + retry

- `parseLabReport` step 1 uses optimistic lock. Duplicate schedules return early.
- Retry cron every 5 min: finds `lab_reports.by_status("analyzing")` older than 10 min, flips to `parse_failed` with `errorCode: "timeout"`.
- User-triggered retry via `retryParseLabReport({ labReportId })` mutation — re-schedules against same row; rate-limit counter NOT re-incremented.

### 3.11 Dev-only admin stub

`convex/admin.ts` mutation `simulateLabUpload({ userId, labOrderId, fixturePath })` — hard-fails on prod deployment via runtime env check. `scripts/seed-lab-upload.ts` drives it for fixture scenarios.

### 3.12 Telemetry

Server logs: `labReportId`, hashed `userId`, `source`, status transitions, marker count, unclassified count, extraction duration, narrative duration, `errorCode`. **Never** PDF content, extracted values, or patient name.

### 4.1 Patient upload (full UI in 2.5)

Route: `app/lab-booking/upload-results.tsx`.

## 8. Architecture invariants (must survive future phases)

1. **Variable-length markers.** `biomarker_reports` renders whatever markers the report contains — 3, 8, or 40. No fixed list, no template narrative.
2. **Source-agnostic pipeline.** Past intake, parse + classify + render branch on nothing derived from `lab_reports.source`. Enforced by `source-agnostic.test.ts`.
3. **Display-only aggregates.** `biomarker_reports` count columns (optimalCount, etc.) are denormalized for dashboard reads; source-of-truth is the sum over `biomarker_values` filtered by report + `deletedAt == null`. Backfill recomputes.
4. **Real-data-only trends.** `useMarkerTrend` never imputes, never synthesizes. Clinical data integrity over UX polish.
5. **Narrative frozen at analysis.** Backfill updates classifications, never narratives, in 2.5. Post-foundation extends to regenerate on unclassified → classified flips.
6. **Pregnancy-sensitive markers never misclassified.** Guard runs before classify. No exception path.
7. **Two registers by design.** Biomarker surfaces use the editorial palette + Instrument Serif / JetBrains Mono. Consultation surfaces use Clinical Luxe + Playfair Display. Do not collapse. See [[decisions/2026-04-17-biomarker-design-register]].
8. **Seed additions are data changes, not code.** Adding a marker = row in `biomarker-ranges.json` + seeder run.
