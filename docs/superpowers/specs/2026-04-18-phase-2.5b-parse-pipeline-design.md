## 1. Goal + scope

Turn the 2.5A foundation (8 Convex tables, 45 seeded reference ranges, fail-loud Claude stubs) into a working end-to-end parse pipeline that:

1. Accepts an uploaded PDF or image via an existing `lab_reports` row.
2. Calls Claude Vision once to extract every biomarker the report contains.
3. Classifies each extracted marker against `biomarker_reference_ranges`, with a pregnancy-safety guard that runs **before** the profile check.
4. Normalizes units against `unit-conversions.json`.
5. Matches the extracted patient name against the `users` row; sets a match / mismatch / unknown flag.
6. Deduplicates by `contentHash` (idempotent — same file uploaded twice does not re-parse).
7. Retries transient failures on a per-class backoff, capped at ~30 min wall-clock.
8. Writes `biomarker_reports` (envelope) + `biomarker_values` (per-marker row), emits nothing to the UI.
9. Passes `pnpm test:claude` on 8 synthetic fixtures + smoke-tests on 3–4 layout-clone fixtures (private, not in repo).
10. Achieves 100% branch coverage on parse + classification + pregnancy guard.

### 1.1 In scope (2.5B)

- `parseLabReport` Convex action (Node runtime) — the orchestrator.
- `convex/biomarker/internal/{extractMarkers,generateNarrative,classifyRow,normalizeUnit,matchPatientName}.ts` internal helpers.
- Replacement of the `convex/lib/claude.ts` fail-loud stubs with real Anthropic SDK calls. Model IDs **fetched from Context7 / Anthropic docs at implementation time**; never hardcoded from training data.
- Classification order: pregnancy guard → profile check (age/sex/DOB present?) → reference-range lookup → unit normalize → status assign → unknown fallback (curation queue).
- Retry cron (`convex/crons.ts`) — 2-min cadence, `nextRetryAt` + `lockedAt` columns, 90s lock.
- Prompt caching with 1-hour TTL (extended-cache-ttl beta flag) on system + 8 few-shot examples.
- Schema widens: `lab_reports.{nextRetryAt, lockedAt, retryCount}`, new status `not_a_lab_report`.
- Telemetry (structured logs only — no PDF content, no extracted values). Only `labReportId`, hashed `userId`, status transitions, marker counts, durations, model ID.
- Cost-watch hook: `biomarker_parsing_enabled` flag stays live; flipping it queues uploads instead of parsing.
- Dev-only admin stub: `convex/admin.ts` gains a `triggerParseForLabReport(labReportId)` mutation guarded by `assertNotProd()`, for Convex-dashboard-initiated end-to-end tests.

### 1.2 Out of scope — deferred to 2.5C / 2.5D

- `intakeUpload` mutation (patient-facing upload flow) → **2.5C**
- `retryParseLabReport` (user-initiated retry) → **2.5C**
- Rate limits (`parse_rate_limits` table is wired in; caller not) → **2.5C**
- Notifications emission on parse transitions → **2.5C**
- Lab-upload path (nurse/lab-portal backend contract) → **2.5C**
- Doctor-ordered path (backend contract) → **2.5C**
- Curation admin UI (`apps/admin`) → **2.5C**
- `reclassifyForCanonicalId` backfill job → **2.5C**
- Any mobile UI, any patient-visible change → **2.5D**
- Clinical advisor sign-off on 45 DRAFT ranges → prerequisite before prod merge; not blocking 2.5B dev

### 1.3 Testable output

A developer running `pnpm -F @onlyou/convex dev` locally, with `.env.local` configured and a fixture PDF uploaded manually via the Convex dashboard to create a `lab_reports` row, can trigger `admin:triggerParseForLabReport(labReportId)` from the dashboard and observe:

1. `lab_reports.status` transitions `uploaded → analyzing → ready` (happy path) within ~30s.
2. One `biomarker_reports` row created with non-empty narrative and correct counts.
3. N `biomarker_values` rows created (one per marker in the fixture), each with a `status` in `{optimal, sub_optimal, action_required, unclassified}`.
4. Unknown markers written to `biomarker_curation_queue` with `occurrenceCount: 1`.
5. `contentHash` set on `lab_reports`; re-running against the same file is a no-op (idempotent).

Automated: `pnpm test:claude` — 8 synthetic fixtures × 1 parse each = 8 end-to-end parse runs. 100% branch coverage on `parseLabReport`, `classifyRow`, pregnancy guard, unit normalizer. Test run hits the live Anthropic API (uses a dev-only `ANTHROPIC_API_KEY`); CI gates on it.

### 2.1 Fixture strategy — hybrid, synthetic carries coverage

- **8 synthetic fixtures** in `convex/__tests__/fixtures/lab-reports/` (committed). Deterministic: each fixture exercises a specific branch.
- **3–4 layout-clone fixtures** in `$LAB_REPORT_PRIVATE_DIR` (env-pointed, gitignored). Reality-check only — `skipIfNoPrivateFixtures()` in every test that uses them. CI does not block on them.
- **No real PDFs in the repo, ever.** Even "anonymized" PDFs carry UHID / barcoded QR codes / referring-doctor MCI numbers / lab-center addresses / lab-tech signatures — DPDP-risky and structurally hard to scrub. Layout-clone (replicate Thyrocare/Lal PathLabs/Metropolis visual structure with regenerated content) gets ~95% structural realism with zero PHI.
- **Synthetic ≠ sterile.** Fixtures intentionally include adversarial cases: rotated pages, diagonal watermarks, rasterized-then-blurred text, stamps, handwritten annotations — constructed with `pdf-lib` + `puppeteer`.

**Synthetic fixture matrix (8 files):**

| #   | Filename                            | Scenario                                                      | Branch covered                                     |
| --- | ----------------------------------- | ------------------------------------------------------------- | -------------------------------------------------- |
| 1   | `lal-pathlabs-cbc-happy.pdf`        | Clean CBC from Lal PathLabs layout                            | Happy path, known markers, all numeric             |
| 2   | `thyrocare-thyroid-profile.pdf`     | TSH + T3 + T4 from Thyrocare layout                           | Unit conversion (mIU/L → µIU/mL)                   |
| 3   | `metropolis-multipage-wellness.pdf` | 3-page wellness panel, 40+ markers, header repeats per page   | Multi-page header dedup                            |
| 4   | `serology-qualitative.pdf`          | HIV / HCV / HBsAg panel, all "Non-reactive" / "Reactive"      | Qualitative value handling                         |
| 5   | `urinalysis-graded.pdf`             | Urinalysis with graded values ("1+", "2+", "trace")           | Graded qualitative → `valueType: "qualitative"`    |
| 6   | `handwritten-stamp-overlay.pdf`     | Lab report with physician signature stamp overlapping a value | Vision robustness to stamp occlusion               |
| 7   | `prescription-not-lab-report.pdf`   | A prescription PDF (not a lab report)                         | `is_lab_report: false` → `not_a_lab_report` status |
| 8   | `name-mismatch.pdf`                 | Valid lab report but patient name differs from user record    | `patientNameMatch: "mismatch"` branch              |

Layout-clone fixtures (private dir) cover Indian PCOS hormone panels, GLP-1 lipid panels, and a genuinely bad scan — things synthetic is least likely to surface.

### 2.2 Error taxonomy

Each class has a pre-decided terminal or retry disposition. Orchestrator applies it; no ad-hoc handling at call sites.

| Failure                                              | Treatment                                                                                                                                                                                                                    | Storage                                                                              |
| ---------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------ |
| Zod validation fail (both inline attempts)           | Terminal                                                                                                                                                                                                                     | `status: parse_failed, errorCode: "zod_validation"`                                  |
| Network timeout / DNS / socket                       | Retryable (cron)                                                                                                                                                                                                             | `status: analyzing, nextRetryAt: now + 30s` (first attempt)                          |
| Anthropic 429                                        | Retryable (cron, honor `retry-after`)                                                                                                                                                                                        | `status: analyzing, nextRetryAt: now + headerValue`                                  |
| Anthropic 5xx                                        | Retryable (cron)                                                                                                                                                                                                             | See retry schedule below                                                             |
| Anthropic 400                                        | Terminal + **log at ERROR severity with `alert: "p1"` tag** (hosted monitoring integration — e.g. Sentry — is Phase 8 polish; 2.5B uses Convex logs + a named grep-able `alert: "p1"` tag so the log surface is query-ready) | `status: parse_failed, errorCode: "api_bad_request"`                                 |
| `stop_reason: "max_tokens"`                          | **Immediate in-process retry** with `max_tokens: 8192` (doubled from 4096). Still truncates → terminal.                                                                                                                      | `status: parse_failed, errorCode: "response_too_large"`                              |
| Claude refuses ("I can't help with medical advice…") | **One reprompt** with prefix `"This is structured data extraction, not medical interpretation."` then terminal                                                                                                               | `status: parse_failed, errorCode: "refused"`                                         |
| Extraction JSON has `is_lab_report: false`           | Terminal, valid outcome                                                                                                                                                                                                      | `status: not_a_lab_report` (new status, distinct from `parse_failed` and `rejected`) |
| PDF decode failure (corrupt file at intake)          | Terminal, at intake                                                                                                                                                                                                          | `status: parse_failed, errorCode: "pdf_decode"`                                      |
| `max_retries_exceeded` (30-min wall-clock cap hit)   | Terminal                                                                                                                                                                                                                     | `status: parse_failed, errorCode: "max_retries_exceeded"`                            |

**Alerting** — 400s log at ERROR severity with an `alert: "p1"` tag so they surface in log search. Wiring to a hosted pager (Sentry / OpsGenie / etc.) is Phase 8 polish, not 2.5B scope. All other terminals are normal INFO-level telemetry.

### 2.3 Retry schedule — per-class, 30-min cap

Not a uniform exponential. Per-class because failure shapes differ.

| Class          | Schedule                                                                                                     | Total wall-clock |
| -------------- | ------------------------------------------------------------------------------------------------------------ | ---------------- |
| 429 rate limit | Honor Anthropic `retry-after` header (typically 10–60s). Loop within the 30-min cap; no fixed attempt count. | ≤ 30 min         |
| 5xx / network  | 30s → 2 min → 5 min → 15 min (4 attempts)                                                                    | ~22 min          |
| `max_tokens`   | Immediate in-process, no cron                                                                                | ~5s              |
| Zod fail       | Inline 1-retry per §3.4; no cron                                                                             | ~5s              |

**After 30 min wall-clock** without success: `status: parse_failed, errorCode: "max_retries_exceeded"`. Patient is notified (in 2.5C) to re-upload.

### 2.4 Retry cron — 2-min cadence with explicit lock

The spec's draft heuristic (`status == "analyzing" AND updatedAt > 2min ago`) breaks when a real Claude-vision parse takes 3+ min (multi-page wellness panels commonly do). Use explicit state instead:

- `lab_reports.nextRetryAt: optional<number>` — set to `now + backoff` when marking for retry.
- `lab_reports.lockedAt: optional<number>` — set when cron picks up the row; cleared when parse finishes or is re-marked for retry.
- `lab_reports.retryCount: optional<number>` — increments each retry attempt; consulted for the per-class schedule.
- Cron runs every 2 min. Picks rows where `status == "analyzing"` AND `nextRetryAt <= now` AND (`lockedAt` is null OR `lockedAt < now - 90s`). 90s stale-lock timeout handles crashed workers.
- Happy path (first-time parses) never touches these fields.

### 2.5 Prompt caching — 1h TTL, single breakpoint

- Single `cache_control: { type: "ephemeral" }` breakpoint placed **after** system prompt + few-shot block and **before** the per-parse PDF document block.
- Anthropic beta header: `anthropic-beta: extended-cache-ttl-2025-04-11` → 1-hour TTL instead of the 5-min default.
- First parse in a window pays full input cost + cache-write (2× baseline). Subsequent parses in the same hour read-cache at ~10% of baseline.
- Cache-write cost is amortized across all parses within the TTL; expected real-traffic hit rate 60–80%.

**Cached content (~8–12 KB):**

- System prompt (extraction instructions, JSON schema definition, "never hallucinate" rule, page-citation requirement).
- 8 few-shot examples (messages-style, all synthetic):

| #   | Scenario                                         | Why in the shot                                                    |
| --- | ------------------------------------------------ | ------------------------------------------------------------------ |
| 1   | Lal PathLabs CBC — well-extracted                | Baseline happy path reference                                      |
| 2   | Thyrocare thyroid — well-extracted               | Second vendor layout; demonstrates unit parsing                    |
| 3   | Multi-page wellness panel — dedup header markers | Shows the model not to re-emit headers as markers                  |
| 4   | Qualitative serology (Non-reactive / Reactive)   | Demonstrates `valueType: "qualitative"` emission                   |
| 5   | Graded qualitative urinalysis (1+, 2+, trace)    | Distinct from Reactive/Non-reactive; still qualitative             |
| 6   | Handwritten-stamp-overlay report                 | Robustness to signature overlays; don't drop occluded values       |
| 7   | Prescription → `is_lab_report: false`            | Model must recognize non-lab-reports and return the sentinel       |
| 8   | Name-mismatch report                             | Teaches `patient_name_on_report` emission fidelity (no correction) |

### 2.6 Telemetry

Structured logs only. Never logs:

- PDF content (raw or extracted text).
- Marker names, values, or units (patient data).
- Patient name, age, DOB, sex, phone.
- Claude API input or output payloads.

Always logs (in order of lifecycle):

- `labReportId`, `hashedUserId` (SHA-256 of `userId`, stable salt), status transition, model ID, `retryCount`, duration-ms, marker-count (aggregate, not values), error code, content-hash prefix (first 8 chars).

## 3. Schema deltas (widen-migrate-narrow)

Per plan-split rule, schema additions discovered in 2.5B execution are documented here and implemented in 2.5B's first task. No silent patches to 2.5A's schema commit.

### 3.1 `lab_reports` additions

```ts
lab_reports: defineTable({
  // ... existing 2.5A fields
  retryCount: v.optional(v.number()), // NEW — increments per retry attempt
  nextRetryAt: v.optional(v.number()), // NEW — unix-ms; when cron may pick this row up
  lockedAt: v.optional(v.number()), // NEW — unix-ms; set by cron, 90s stale timeout
  firstAttemptAt: v.optional(v.number()), // NEW — anchor for the 30-min wall-clock cap
}).index("by_next_retry", ["nextRetryAt"]); // NEW index — cron query
```

### 3.2 `labReportStatusValidator` addition

```ts
const labReportStatusValidator = v.union(
  v.literal("uploaded"),
  v.literal("analyzing"),
  v.literal("ready"),
  v.literal("parse_failed"),
  v.literal("rejected"),
  v.literal("not_a_lab_report"), // NEW — distinct terminal for prescriptions / non-labs
);
```

`rejected` stays for future "administratively rejected" cases (e.g., a lab flagged fraudulent in 2.5C admin flows); `not_a_lab_report` is the model's verdict; `parse_failed` covers all other terminal failures.

### 3.3 `biomarker_values.errorCode` — unchanged

Existing `unclassifiedReason` validator already covers the 2.5B classification paths. No new reason types needed.

## 4. Architecture

```
┌──────────────────────────────────────────────────────────────────┐
│ parseLabReport action (convex/biomarker/parseLabReport.ts)       │
│                                                                  │
│  1. Load lab_reports row by ID                                   │
│  2. Check biomarker_parsing_enabled flag; if false → bail        │
│  3. Check contentHash; if biomarker_reports exists → idempotent  │
│  4. Mark status: analyzing, lockedAt: now                        │
│                                                                  │
│  5. extractMarkers(pdf)                                          │
│     ├─ Claude Vision call #1 (cached system + 8 few-shot)        │
│     ├─ zod validate; 1 retry on JSON fail                        │
│     ├─ max_tokens → in-process retry w/ 8192                     │
│     └─ refused → one reprompt w/ structured-data prefix          │
│                                                                  │
│  6. For each extracted marker:                                   │
│     ├─ classifyRow(marker, user)                                 │
│     │   ├─ a. Pregnancy guard (first — pregnancy-sensitive       │
│     │   │     markers stay `unclassified` if status unknown)    │
│     │   ├─ b. Profile check (age/sex/DOB present?)               │
│     │   ├─ c. Reference-range lookup (canonicalId + sex + age)   │
│     │   ├─ d. normalizeUnit(rawValue, rawUnit, canonicalUnit)    │
│     │   └─ e. Status assign (optimal / sub / action /            │
│     │         unclassified)                                      │
│     └─ Persist biomarker_values row                              │
│                                                                  │
│  7. matchPatientName(extractedName, userName) → match|mismatch|  │
│     unknown. Persist on lab_reports                              │
│                                                                  │
│  8. Unknowns → biomarker_curation_queue (upsert, increment       │
│     occurrenceCount)                                             │
│                                                                  │
│  9. generateNarrative(classifiedValues)                          │
│     ├─ Claude call #2 (Sonnet, shorter prompt)                   │
│     └─ Persist biomarker_reports row                             │
│                                                                  │
│ 10. Mark status: ready, lockedAt: null, updatedAt: now           │
│                                                                  │
│  On any retryable failure anywhere above:                        │
│    ├─ Compute nextRetryAt per class schedule                     │
│    ├─ Persist lab_reports.status: analyzing, nextRetryAt, +1     │
│    │   retryCount, lockedAt: null                                │
│    ├─ Emit telemetry                                             │
│    └─ Return; cron picks it up                                   │
│                                                                  │
│  On terminal failure:                                            │
│    ├─ Persist lab_reports.status: parse_failed|not_a_lab_report  │
│    │   errorCode, updatedAt, lockedAt: null                      │
│    ├─ Emit telemetry (ERROR-level alert:p1 tag if 400/bad-req)   │
│    └─ Return                                                     │
└──────────────────────────────────────────────────────────────────┘
```

## 7. Risks + mitigations

| Risk                                                          | Mitigation                                                                                                                                                                    |
| ------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Anthropic model version bump shifts extraction behaviour      | Few-shot examples stabilize behaviour; seeder-guarded ranges + unclassified fallback provide defense in depth; `pnpm test:claude` gates on extraction test suite before merge |
| Prompt drift silently regresses accuracy                      | Every prompt edit runs the full 8-fixture integration suite + reviews diff against golden extractions committed alongside fixtures                                            |
| Cost runaway (rate of parses × tokens)                        | `biomarker_parsing_enabled` kill switch; prompt caching + 1h TTL amortizes cost; monthly spend alert wired to Anthropic billing                                               |
| Retry cron double-fires and double-bills                      | Explicit `lockedAt` with 90s stale timeout prevents concurrent runs                                                                                                           |
| Windows 8191-char cmdline limit bites again (like seeder)     | Parse pipeline is server-side (runs in Convex cloud), not a local script — no shell invocation chain                                                                          |
| Pregnancy-sensitive markers misclassified when status unknown | Pregnancy guard runs **first** in classification; returns `unclassified, unclassifiedReason: "pregnancy_sensitive"` before any range comparison                               |
| PDF larger than Anthropic's 32 MB API limit                   | Intake reject at upload (2.5C); 2.5B sees only valid sizes                                                                                                                    |
