## 1. Goal + scope

Make the parse pipeline usable by real patients end-to-end. Ship every backend capability needed for 2.5D (mobile UI) to run against: real upload → parse → classify → degrade-gracefully-or-auto-classify unknown markers → notify → reclassify on range changes. No user interface in this plan.

### 1.1 In scope (2.5C)

**Real ingestion**

- `intakeUpload` mutation — patient-triggered upload entry point with rate limits (5/day + 50/month, IST calendar), size + mime guards, atomic rate-limit-bucket + `lab_reports` insert
- `retryParseLabReport` mutation — user-triggered retry capped at 3 lifetime attempts per lab_report; distinct from 2.5B's automatic retry cron
- Shared `createLabReport` internal helper — single source of truth for `lab_reports` row creation; consumed by all three ingestion surfaces (patient, lab partner, nurse-flow dev stub)

**Auto-classification**

- Fuzzy alias matcher layered over 2.5B's exact-alias path (Jaro-Winkler ≥ 0.92 AND unit-match AND Claude `canonical_id_guess` agreement)
- Panel-code auto-skip — regex-filter obvious lab panel codes (e.g., `HEM-401`, `CBC/PLT-RATIO`), mark queue row `wont_fix` inline
- **Auto-DRAFT range generation** — after an unknown marker crosses a cooling-off threshold of ≥3 occurrences AND Claude proposes a canonical fitting one of the 8 known categories (Nutrient Health, Thyroid, Metabolic, CBC, Hormonal Balance, Lipids, Liver, Kidney), the system runs a secondary Claude call to propose DRAFT thresholds, writes a `biomarker_reference_ranges` row with `clinicalReviewer: "DRAFT — auto-generated YYYY-MM-DD"` and `autoGenerated: true`, then triggers `reclassifyForCanonicalId` to classify pending unclassified values for that marker. Prod-deploy guard unchanged: any DRAFT row (seeded or auto) blocks prod until clinician sign-off. See §6 for the full gate.

**Reclassification**

- `reclassify_locks` table + `reclassifyForCanonicalId` action — serialized per-canonical; triggered automatically by range create/update/deactivate/reactivate and by curation resolve
- `reclassifyAllReports({ mode: "preview" | "commit" })` action — global reclassify for the DRAFT→reviewed range swap; preview returns coupled payload (totalValues / wouldChange / byTransition / byMarker / affectedReports / affectedPatientCount / rangesSignature); commit verifies `rangesSignature` against current state before writes
- `admin_audit_log` table (writes only in 2.5C; viewer UI is Phase 5) — records every range CRUD, curation resolve, reclassify preview + commit

**Notifications**

- Pipeline emits all four `notifications` kinds on their terminal transitions: `lab_report_ready`, `lab_report_parse_failed`, `lab_report_updated`, `lab_report_uploaded_for_you`
- Band-change-only emission: `lab_report_updated` fires only when ≥1 biomarker_value actually flipped status or unclassifiedReason within a given report

**Portal contracts (stubs)**

- `labUploadResult` mutation — real body shipping in 2.5C; env-flag gated at runtime (throws `endpoint_disabled` when `LAB_PORTAL_ENABLED !== "1"`); additional prod guard throws `endpoint_disabled_unsafe_in_prod` when `LAB_PORTAL_REAL_AUTH !== "1"` on production deployments. Phase 6 sets both when partner auth ships.
- `biomarkerReportsForPatient` query — gated the same way with `DOCTOR_PORTAL_ENABLED` + `DOCTOR_PORTAL_REAL_AUTH`. Phase 4 wires the doctor portal against this contract.
- `simulateLabUpload` admin dev-stub — admin-role + prod runtime check (same pattern as 2.5B's `triggerParseForLabReport`); exercises `lab_upload` AND `nurse_flow` source values via the shared helper

**Carry-forwards from 2.5B review**

- I-3: 8 few-shot extraction examples — 3 existing golden JSONs + 5 synthetic founder-verified. Unblocks `FEW_SHOT_EXAMPLES` empty-array from 2.5B.
- M-2: drop `"classify_row"` literal from `ParseEvent` union in `convex/lib/telemetry.ts`
- M-5/M-6/M-7: lint hygiene and typing polish

**Tests**

- Automated test suite (§9) across 10+ new Convex test files
- Manual Convex dashboard E2E against a real fixture PDF via `intakeUpload` — hard gate before 2.5D enters its own approval review

```ts
userRetryCount: v.optional(v.number()),  // lifetime cap at 3; enforced in retryParseLabReport
```

**`biomarker_values`**

```ts

// New index:
```

Rationale: the curation detail view in Phase 5 (and the auto-classification triggers in 2.5C) need a direct index lookup from queue row to its contributing values. Keyed on the same `normalizedKey` the parser already computes when upserting the queue row. See Section 4 of the brainstorm tightening.

**`biomarker_reference_ranges`**

```ts

```

Clinician-review workflow can filter by this flag to identify rows needing extra scrutiny.

**`lab_orders.status`** — widen the union to add `"failed"`

```ts
v.union(
  v.literal("draft"),
  v.literal("sent_to_lab"),
  v.literal("awaiting_results"),
  v.literal("results_uploaded"),
  v.literal("cancelled"),
  v.literal("failed"), // NEW — lab rejected / sample lost / panel unavailable

```

No caller in 2.5C transitions into `failed`; Phase 6 lab portal and admin overrides use it. Shipping the literal now avoids a widening migration later.

### 3.2 New tables

**`reclassify_locks`**

```ts
defineTable({
  canonicalId: v.string(), // sentinel "*" for global reclassify
  ownerToken: v.string(), // UUID per action start; release verifies match
  acquiredAt: v.number(),
  expiresAt: v.number(), // acquiredAt + 5 minutes
  action: v.union(
    v.literal("reclassifyForCanonicalId"),
    v.literal("reclassifyAllReportsPreview"),
    v.literal("reclassifyAllReportsCommit"),
  ),
})
  .index("by_canonical", ["canonicalId"])
  .index("by_expires", ["expiresAt"]);
```

Serializes reclassify actions. Acquire-or-fail semantics: second caller on the same canonicalId (or the global `"*"`) receives `reclassify_in_progress`. Stale-lock cron runs every 5 min, deletes rows where `expiresAt < now`; normal release happens by `(canonicalId, ownerToken)` match.

**`admin_audit_log`**

```ts
defineTable({
  adminUserId: v.id("users"),
  action: v.union(
    v.literal("curation_resolve"),
    v.literal("curation_wont_fix"),
    v.literal("range_create"),
    v.literal("range_update"),
    v.literal("range_deactivate"),
    v.literal("range_reactivate"),
    v.literal("reclassify_canonical_commit"),
    v.literal("reclassify_all_preview"),
    v.literal("reclassify_all_commit"),
  ),
  targetTable: v.union(
    v.literal("biomarker_curation_queue"),
    v.literal("biomarker_reference_ranges"),
  ),
  targetId: v.string(), // document ID as string (union-generic)
  before: v.optional(v.any()), // row snapshot before mutation; null on create
  after: v.optional(v.any()), // row snapshot after mutation; null on delete
  timestamp: v.number(),
})
  .index("by_admin", ["adminUserId", "timestamp"])
  .index("by_target", ["targetTable", "targetId", "timestamp"]);
```

Write-only in 2.5C. Convex dashboard is the only read surface until Phase 5. All 2.5C actions that mutate reference ranges, curation queue, or trigger reclassify commits write a row here in the same transaction as the data change.

## 4. Wave 1 — Real ingestion

### 4.1 `intakeUpload` mutation

**Surface:** `convex/biomarker/intakeUpload.ts`

**Args:**

```ts
{
  fileId: v.id("_storage"),
  mimeType: v.union(
    v.literal("application/pdf"),
    v.literal("image/jpeg"),
    v.literal("image/png"),
  ),
  fileSizeBytes: v.number(),
  source: v.literal("patient_upload"),  // hard-pinned; lab/nurse enter via separate mutations
  labOrderId: v.optional(v.id("lab_orders")),
}
```

**Flow (all within one Convex mutation = one transaction):**

1. Auth check — session's `userId` must match; reject `unauthenticated` otherwise
2. Kill-switch — read `biomarker_parsing_enabled` feature flag; if false, reject `parsing_disabled`
3. Size gate — `fileSizeBytes > 20 * 1024 * 1024` → `file_too_large`
4. Mime gate — rejected unions never reach here, but double-check to be paranoid
5. Rate-limit read — `parse_rate_limits.by_user_window(userId, "day", IST_DAY_BUCKET)` AND `by_user_window(userId, "month", IST_MONTH_BUCKET)`. If either count ≥ limit (5 / 50) → `{ error: "rate_limited", which: "day" | "month", resetAt: <next midnight IST> }`
6. Insert `lab_reports` with `status: "uploaded"`, `userRetryCount: 0`, `retryCount: 0`
7. Upsert both `parse_rate_limits` buckets (+1 each)
8. `scheduler.runAfter(0, "parseLabReport", { labReportId })`
9. Return `{ labReportId, status: "uploaded" }`

**Atomicity guarantee:** Convex mutations are transactional. Steps 6–8 commit atomically. No crash window between bucket increment and lab_report insert.

### 4.2 `retryParseLabReport` mutation

**Surface:** `convex/biomarker/retryParseLabReport.ts`

**Args:** `{ labReportId: v.id("lab_reports") }`

**Guards:**

1. Auth — session owns this lab_report
2. `lab_reports.status === "parse_failed"` — otherwise `invalid_retry_state`
3. `userRetryCount < 3` — otherwise `retry_cap_exceeded` with terminal UX copy "We can't process this report. Try uploading again or contact support."

**Reset semantics (deliberate):**

| Field                                         | Action       | Why                                              |
| --------------------------------------------- | ------------ | ------------------------------------------------ |
| `userRetryCount`                              | Bump +1      | Lifetime cap enforcement                         |
| `retryCount`                                  | Reset to 0   | New auto-retry budget for this attempt           |
| `errorCode` / `errorMessage`                  | Clear        | Stale error shouldn't shadow the new run         |
| `nextRetryAt` / `lockedAt` / `firstAttemptAt` | Clear        | Cron state for the previous run                  |
| `status`                                      | `"uploaded"` | Re-enters the pipeline                           |
| `parse_rate_limits`                           | Not touched  | Retries don't re-consume the daily/monthly quota |

Schedule `parseLabReport` immediately. Return `{ labReportId, status: "uploaded", userRetryCount }`.

### 4.3 Shared `createLabReport` internal helper

**Surface:** `convex/biomarker/lib/createLabReport.ts`

Signature:

```ts
export async function createLabReport(
  ctx,
  {
    userId,
    source, // "patient_upload" | "lab_upload" | "nurse_flow"
    labOrderId,
    fileId,
    mimeType,
    fileSizeBytes,
  },
): Promise<{ labReportId: Id<"lab_reports"> }>;
```

Responsibilities:

- Size + mime validation
- Insert `lab_reports` row
- If `source ∈ { lab_upload, nurse_flow }` AND `labOrderId` supplied → link it on `lab_orders.labReportId` and flip `lab_orders.status` to `"results_uploaded"`
- Schedule `parseLabReport`
- Return `labReportId`

Callers wrap with surface-specific guards:

- `intakeUpload` → session auth + rate-limit check/increment
- `labUploadResult` → partner token + order-state check (§8)
- `simulateLabUpload` (admin dev-stub) → admin role + prod runtime guard (§8)

Rate limiting is NOT inside this helper — patient-upload-only concern.

## 6. Wave 3 — Smart unknowns (the heart of this plan)

Classification pipeline in 2.5B has one path for known aliases + a fall-through to `unclassified:not_in_reference_db` that upserts the curation queue. Wave 3 inserts three new layers between those:

```
Claude extraction → normalized name
  │
  ├─ [1] Exact alias match?              → classify, done (existing 2.5B path)
  │
  ├─ [2] Fuzzy alias match ≥0.92 AND     → classify as exact-match, mark confidence
  │      unit AND canonical_id_guess
  │      all agree?
  │
  ├─ [3] Panel-code pattern match?       → upsert queue row, mark status="wont_fix",
  │                                        record reason="auto_panel_code";
  │                                        NO biomarker_value row (skip)
  │
  ├─ [4] Cooling-off threshold met       → run secondary Claude call to propose
  │      (normalizedKey occurrence ≥3)     DRAFT range in a known category;
  │      AND Claude proposes a             on success, write range row with
  │      canonical in the category           autoGenerated=true, trigger
  │      whitelist?                        reclassifyForCanonicalId;
  │                                        current value classifies immediately
  │
  └─ [5] None of the above                → upsert queue row with status="pending";
                                           biomarker_value remains unclassified
                                           (existing 2.5B behaviour)
```

### 6.1 Fuzzy alias matcher

**Surface:** `convex/biomarker/lib/fuzzyAliasMatch.ts`

Pure function. Inputs: normalized `name_on_report`, `raw_unit`, Claude's `canonical_id_guess`, full reference-range alias list (cached).

Algorithm:

1. For each known canonicalId in the active ranges, compute `max(jaroWinkler(normalizedName, alias))` across its aliases
2. Find the best canonical match — requires score ≥ 0.92
3. Require unit compatibility — `rawUnit` either matches `canonicalUnit` directly OR has a conversion in `unit-conversions.json`
4. Require agreement — Claude's `canonical_id_guess` (if non-null) must equal the best-match canonical OR be null (null = no guess = don't block)
5. All three pass → return best-match canonicalId. Otherwise → null

Returns null on ambiguity (e.g., two canonicals tie at 0.93); those fall through to the queue. Ambiguity in unit (e.g., raw unit has no conversion row) also returns null.

### 6.2 Panel-code auto-skip

**Surface:** `convex/biomarker/lib/panelCodeDetect.ts`

Regex battery against `nameOnReport`. Any match → skip the marker entirely (no `biomarker_value` row), upsert a curation queue row with `status: "wont_fix"`, `resolvedByUserId: null` (system-resolved), `reason: "auto_panel_code"`.

Patterns:

- `/^[A-Z]{2,4}[-\s]?\d{2,}$/` — e.g., `HEM-401`, `LAB-72`, `CBC 401`
- `/^[A-Z]+\/[A-Z]+[-\s]?RATIO$/i` — e.g., `CBC/PLT-RATIO`
- `/(PANEL|CODE|PROFILE)[-\s]?\d+/i`

Configurable: list in `packages/core/seeds/panel-code-patterns.json`, regenerable. Each pattern has a human-readable comment.

### 6.3 Auto-DRAFT range generation — **the critical gate**

Triggered when:

1. An unknown marker (not exact match, not fuzzy match, not panel code) reaches the classifier
2. AND the curation queue row for its `normalizedKey` has `occurrenceCount ≥ 3` (cooling-off threshold)
3. AND Claude's `canonical_id_guess` is non-null

**Secondary Claude call** (separate from the main extraction call, on the narrative model to leverage cheaper pricing):

```
Input (text-only, no PDF):
{
  "name_on_report": "Aspartate Aminotransferase",
  "raw_unit": "U/L",
  "lab_printed_range": "5-40",
  "canonical_id_guess": "ast_sgot",
  "categories_allowed": ["Nutrient Health", "Thyroid", "Metabolic", "CBC",
                         "Hormonal Balance", "Lipids", "Liver", "Kidney"]
}

Required output:
{
  "is_real_biomarker": boolean,
  "canonical_id": string | null,        // preferred canonical (may differ from guess)
  "display_name": string | null,
  "aliases": string[],                   // other names this marker goes by
  "category": string | null,             // must be in categories_allowed
  "canonical_unit": string | null,
  "sex": "male" | "female" | "any" | null,
  "age_min": number | null,
  "age_max": number | null,
  "pregnancy_sensitive": boolean | null,
  "optimal_min": number | null,
  "optimal_max": number | null,
  "sub_optimal_below_min": number | null,
  "sub_optimal_above_max": number | null,
  "action_below": number | null,
  "action_above": number | null,
  "explainer": string | null,            // ~1–2 plain-English sentences
  "source_citation": string | null,      // Claude cites a standard medical reference
  "confidence": number                   // 0.0–1.0
}
```

**Gate: write the range row only when ALL of these pass:**

1. `is_real_biomarker === true`
2. `category ∈ categories_allowed`
3. `confidence ≥ 0.75`
4. All threshold numeric fields non-null
5. Threshold ordering passes the existing `seed-validation` check (`actionBelow < subOptimalBelowMin < optimalMin ≤ optimalMax < subOptimalAboveMax < actionAbove`)
6. `source_citation` non-empty

If any fails → fallback to the existing queue-only path (today's behaviour). Queue row stays `pending`.

If all pass → write `biomarker_reference_ranges` row with:

```ts
{
  ...claudeOutput,
  clinicalReviewer: `DRAFT — auto-generated ${YYYY-MM-DD}`,
  reviewedAt: undefined,
  isActive: true,
  autoGenerated: true,
  updatedAt: now,
}
```

Then:

- Mark the queue row `resolved` with `resolvedReferenceRangeId` pointing at the new row
- Trigger `reclassifyForCanonicalId(canonical_id)` so the current marker AND any prior unclassified values for the same normalizedKey classify immediately
- Write an audit log entry with `action: "range_create"`, `adminUserId: systemUserId` (a sentinel ID for system actions)

### 6.4 Why these gates make the hallucination risk acceptable

| Guard                         | What it catches                                                                     |
| ----------------------------- | ----------------------------------------------------------------------------------- |
| Cooling-off (≥3 occurrences)  | Claude's one-off misreads never become ranges; only markers that keep appearing     |
| Category whitelist            | Claude can't invent ranges for unrelated lab fields it misinterprets as biomarkers  |
| Confidence ≥ 0.75             | Low-confidence guesses fall to queue, not rangetable                                |
| Threshold-ordering validation | Mathematically impossible threshold sets are rejected                               |
| `source_citation` required    | Claude must claim a real reference (clinician verifies during DRAFT review)         |
| DRAFT-pending-review flag     | Prod deploy is blocked until the clinician reviews every row, auto-generated or not |
| `autoGenerated: true` tag     | Clinician-review workflow can filter and scrutinize auto-rows specifically          |
| Convex dashboard visibility   | Founder can eyeball auto-rows anytime during dev                                    |

Anything that gets past all of these is unlikely to cause clinical harm even in dev, and is guaranteed to be caught before prod by the DRAFT-sign-off gate.

### 6.5 What this does to curation queue volume

Expected steady state after 2.5C ships:

- ~80% of markers in real Indian lab reports already covered by the 45-row seed → exact or alias match
- ~10% covered by fuzzy match on variants and typos
- ~5% caught by panel-code auto-skip
- ~3% auto-DRAFT-generated (real but novel markers hitting the cooling-off threshold)
- ~2% land in the curation queue as `pending` for human review

At expected early Phase 3 volume (single-digit daily uploads), the `pending` queue accumulates maybe 1–3 rows per week. Non-urgent. Phase 5 admin UI handles triage when volume grows.

## 7. Wave 4 — Reclassify engine

### 7.1 `reclassifyForCanonicalId` action

**Surface:** `convex/biomarker/reclassifyForCanonicalId.ts`

Triggered automatically by:

- `createReferenceRange` / `updateReferenceRange` / `deactivateReferenceRange` / `reactivateReferenceRange` (currently called only via Convex dashboard in 2.5C; Phase 5 wires admin UI callers)
- Curation resolve flows (current callers: auto-range generation from §6.3; future callers: Phase 5 admin UI via `resolveCurationRow` / `resolveCurationWithNewRange`)

Flow:

1. Acquire `reclassify_locks` row with `{ canonicalId, ownerToken: uuid(), acquiredAt: now, expiresAt: now + 5min, action: "reclassifyForCanonicalId" }`. If a row already exists for this canonicalId OR a global `"*"` lock is held → throw `reclassify_in_progress`
2. Query `biomarker_values.by_canonical_status(canonicalId, *)` — all non-deleted values for this canonicalId regardless of current status (range deactivation might flip optimal → unclassified)
3. Iterate in chunks of 100, 50ms sleep between chunks
4. For each value: re-run `classifyRow` against current active ranges (filter `isActive === true`). Collect `changedValueIds` where status OR unclassifiedReason changed
5. Per affected `biomarker_report`, recompute aggregate counts (optimal / sub_optimal / action_required / unclassified). Bump `lastReclassifiedAt`
6. For each affected report with ≥1 changed value, emit one `lab_report_updated` notification
7. Release lock via `(canonicalId, ownerToken)` delete
8. Write audit log: `{ action: "reclassify_canonical_commit", targetTable: "biomarker_reference_ranges", targetId: canonicalId, after: { changedCount, affectedReports } }`

Narrative is NEVER regenerated (invariant #5 from master spec).

### 7.2 `reclassifyAllReports` action — preview and commit

**Surface:** `convex/biomarker/reclassifyAllReports.ts`

Primary use case: the DRAFT→reviewed range swap when the clinician returns edits. See [[decisions/2026-04-20-proceeding-on-draft-ranges]].

**Preview mode:**

```ts
reclassifyAllReports({ mode: "preview" });
```

1. Acquire global lock `(canonicalId: "*", ownerToken: uuid(), action: "reclassifyAllReportsPreview")`
2. Read ALL non-deleted `biomarker_values` once (server-side materialization; ~tens of thousands of rows at MVP scale — acceptable)
3. Compute `rangesSignature` = `MAX(updatedAt)` across all `biomarker_reference_ranges` rows
4. For each value: compute what `classifyRow` WOULD produce. Do not write.
5. Aggregate into:

```ts
{
  totalValues: number,
  wouldChange: number,
  byTransition: Record<string, number>,    // dynamic keys "oldStatus_to_newStatus", only present if count>0
  byMarker: Array<{ canonicalId, displayName, changeCount }>,   // sorted desc
  affectedReports: number,
  affectedPatientCount: number,             // unique userId across affected reports
  rangesSignature: string,                  // the MAX(updatedAt) captured in step 3
  previewedAt: number,
}
```

6. Write audit log `{ action: "reclassify_all_preview", before: null, after: { ...payload without byMarker } }`
7. Release lock, return payload

**Commit mode:**

```ts
reclassifyAllReports({ mode: "commit", rangesSignature });
```

1. Read current `MAX(updatedAt)` on `biomarker_reference_ranges`. If mismatches the caller's `rangesSignature` → throw `ranges_changed_since_preview`
2. Acquire global lock (same as preview)
3. Iterate all non-deleted `biomarker_values` in chunks of 500, 50ms sleep
4. For each value, re-run `classifyRow`; write updates for those whose status or unclassifiedReason changed
5. Recompute aggregates per affected `biomarker_report`; emit `lab_report_updated` per report with ≥1 change
6. Write audit log `{ action: "reclassify_all_commit", before: null, after: { changedValuesCount, affectedReports, duration_ms, rangesSignature } }`
7. Release lock, return payload (same shape as preview plus `committedAt`)

Per [[decisions/2026-04-20-reclassify-preview-payload-coupling]]: all preview fields derive from one snapshot; staleness invalidates the entire payload.

### 7.3 Concurrency guarantees

| Running action            | Blocks                                 | Error returned           |
| ------------------------- | -------------------------------------- | ------------------------ |
| Per-canonical lock on `X` | Another per-canonical on `X`           | `reclassify_in_progress` |
| Global lock `"*"`         | Any per-canonical AND any other global | `reclassify_in_progress` |
| Per-canonical on `X`      | Global lock attempts                   | `reclassify_in_progress` |

Stale-lock cron (`crons.ts`) runs every 5 min, deletes rows where `expiresAt < now`. Expected only as a safety net — normal release uses `(canonicalId, ownerToken)` match.

## 8. Wave 5 — Portal contracts

### 8.1 Runtime flag-gate pattern (two env vars per portal)

```ts
// convex/lib/portalGates.ts
export function assertPortalEnabled(
  portal: "LAB" | "DOCTOR",
  deployment: string,
): void {
  const enabled = process.env[`${portal}_PORTAL_ENABLED`] === "1";
  if (!enabled) throw new ConvexError({ code: "endpoint_disabled" });

  const prodLike = /prod/i.test(deployment);
  const realAuth = process.env[`${portal}_PORTAL_REAL_AUTH`] === "1";
  if (prodLike && !realAuth) {
    throw new ConvexError({ code: "endpoint_disabled_unsafe_in_prod" });
  }
}
```

Both portal handlers call this as the first statement. Dev environments can set `ENABLED=1` alone and exercise the stub; prod requires both `ENABLED=1` AND `REAL_AUTH=1`. Phase 4 and Phase 6 set `REAL_AUTH=1` when real authentication ships.
