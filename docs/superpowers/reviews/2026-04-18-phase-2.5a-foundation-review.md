### Important

**I-1 ŌĆö ESLint rule does not cover the `@onlyou/core` barrel path**
File: `D:/onlyou2-phase-2.5a/apps/mobile/eslint.config.js:12-25`
File: `D:/onlyou2-phase-2.5a/packages/core/src/index.ts:1` (`export * from "./tokens"`)

The rule targets `name: "@onlyou/core/tokens"`, but the `@onlyou/core` top-level barrel also re-exports everything from `./tokens`, including `biomarkerPalette` and `biomarkerFonts`. A well-intentioned developer writing `import { biomarkerPalette } from "@onlyou/core"` in a non-biomarker file would bypass both `no-restricted-imports` blocks silently. Same escape hatch exists for `colors` inside biomarker paths.

The guardrail claim is "ESLint rule blocks cross-register imports both ways" ŌĆö that should cover every resolvable path to those symbols, not just the most common one. Two paths exist (`@onlyou/core` and `@onlyou/core/tokens`); right now only one is guarded.

Recommended fix ŌĆö add a second `paths` entry covering the barrel, in both blocks:

```js
paths: [
  {
    name: "@onlyou/core/tokens",
    importNames: ["biomarkerPalette", "biomarkerFonts"],
    message: "...",
  },
  {
    name: "@onlyou/core",
    importNames: ["biomarkerPalette", "biomarkerFonts"],
    message: "...",
  },
],
```

Small change; keeps the promise the decision doc made.

### Minor

**M-1 ŌĆö `assertNotProd` discards its `ctx` parameter**
File: `D:/onlyou2-phase-2.5a/convex/admin.ts:17-25`

`export function assertNotProd(_ctx: MutationCtx | ActionCtx): void` ŌĆö the `ctx` is marked unused and the check reads `process.env.CONVEX_DEPLOYMENT` only. That works today because `CONVEX_DEPLOYMENT` is reliably set by Convex at runtime. But the signature implies `ctx` carries deployment info, and a future maintainer may reasonably switch to reading it from `ctx.runQuery(api.deployment.info)` or a server-side helper. Either:

- Drop the `ctx` param entirely (it's not used) and rename to `assertNotProd()`, OR
- Use `ctx` to query a server-side deployment indicator (more robust against env-var spoofing in dev shells).

The current shape is a mild lie about dependencies ŌĆö no bug, just a shape that invites confusion. Given this file is a scaffold that 2.5B/C will flesh out, picking one shape now saves refactoring every admin operation later.

**M-2 ŌĆö Prod-pattern regex `prod$` matches more than intended**
File: `D:/onlyou2-phase-2.5a/convex/admin.ts:15`
File: `D:/onlyou2-phase-2.5a/scripts/seed-biomarker-ranges.ts:108`

`PROD_DEPLOYMENT_PATTERNS` / `PROD_PATTERNS` both include `/prod$/i` alongside `/^prod$/i` and `/^production$/i`. The middle pattern matches any deployment name ending in "prod" ŌĆö e.g., `onlyou-staging-not-prod`, `dev-before-prod`, `preprod`, `qa-preprod`, etc. That's over-matching: those are NOT the production deployment, and blocking admin operations there is either a false positive (annoying) or, worse, a false sense of security ("we thought the pattern was `production` exactly, but any dev deployment containing 'prod' would also be blocked").

Two read paths:

- If the **intent** is "only the literal `prod` or `production` deployment", tighten to just `/^prod$/i` and `/^production$/i` and delete `/prod$/i`.
- If the **intent** is defence-in-depth (a dev deployment accidentally named with `prod` suffix should err on the safe side), keep the pattern but document it in the file header with an example.

Right now the regexes and the intent don't commit to either reading. Either is fine; pick one.

**M-3 ŌĆö `seeds/README.md` documents the relaxed threshold invariant, but the prose precedes the reason**
File: `D:/onlyou2-phase-2.5a/packages/core/seeds/README.md:24-27`

The README says the action bounds use `Ōēż` because "the seed collapses action and sub-optimal thresholds on the lower/upper edge ŌĆö both values may be equal." That's accurate but readers arriving cold won't know why the seed collapses them. One extra sentence ŌĆö "For markers where the sub-optimal-to-action boundary is clinically ambiguous (e.g., Vitamin D deficiency starts at <20 regardless of whether you call it 'sub-optimal' or 'action required'), the seed sets `actionBelow == subOptimalBelowMin`" ŌĆö would head off the next reviewer's "wait, why isn't this strict?" question.

Not load-bearing; README hygiene only.

### Suggestions

**S-1 ŌĆö Consider a typecheck-level assertion that `biomarkerFonts` keys match actual loaded font keys**

`biomarkerFonts.display = "InstrumentSerif_400Regular"` is a plain string literal. The `useFonts` map in `_layout.tsx` uses the matching key. If someone renames one side and not the other, typescript will not catch it ŌĆö only a device render will reveal the broken text. A lightweight fix at 2.5D wiring time: export a `const REQUIRED_BIOMARKER_FONTS = [...] as const satisfies readonly string[]` from `biomarker.ts`, import it in `_layout.tsx`, and build the `useFonts` argument from it. Out of scope for 2.5A; note for 2.5D.

**S-2 ŌĆö `convex/lib/claude.ts` could export its sentinel more loudly**

`MODEL_EXTRACTION = "VERIFY_AT_IMPLEMENTATION_PLAN_2_5B"` is already a good trap. A second-line belt-and-braces option: `if (MODEL_EXTRACTION.startsWith("VERIFY_")) throw new Error(...)` inside `extractMarkers` and `generateNarrative` before the "not wired yet" throw. That way, if 2.5B wires the Anthropic client but forgets to update the constant, the fail mode is an explicit message rather than an opaque 400 from the API. The current throw already blocks invocation, so this is marginal ŌĆö suggesting for 2.5B, not required now.
