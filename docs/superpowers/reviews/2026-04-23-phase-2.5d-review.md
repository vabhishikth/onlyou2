#### I-1 · `BiomarkerCard` divide-by-zero when `b.prev === 0` — unguarded in the card while the same math is guarded in `DetailHero`

**File:** `apps/mobile/src/components/biomarker/BiomarkerCard.tsx:110`

```ts
const delta = ((b.value - b.prev) / b.prev) * 100;
```

`DetailHero.tsx:92` guards this exact calculation (`b.prev !== 0 ? … : 0`). `BiomarkerCard` does not. Today the mock data always has `prev > 0` and the hook transform sets `prev = numericValue` (placeholder), so the bug is latent. But the Phase 2.5E spec explicitly wires a real `prev` (prior-report join) — as soon as a patient has a first-ever report with no prior, `prev` will be `0` or `undefined` and `delta` will be `Infinity`, which renders as the literal string `"+Infinity%"`.

DEFERRED lists this as a "Wave 5" item that did not ship in Wave 5. Since the fix is one line and the 2.5E ticket is within touching distance, I recommend closing it here rather than letting a demo break later:

```ts
const delta = b.prev !== 0 ? ((b.value - b.prev) / b.prev) * 100 : 0;
```

Why it matters: the first bad impression is the one the founder will remember. The same math was already hardened in the sibling component — this is trivially fixable in the same branch.

#### I-2 · `AreaChart` x-axis labels iterate over 7 hardcoded slots but the `data` array is now variable-length — first-report patients will see `6mo`, `5` for a single real point

**File:** `apps/mobile/src/components/biomarker/viz/AreaChart.tsx:143,270-282`

```ts
const labels = ["6mo", "5", "4", "3", "2", "1", "now"];
…
{labels.map((label, i) => (
  <Text … x={toX(i)} … >{label}</Text>
))}
```

`toX(i)` divides by `data.length - 1 || 1`. When `data.length === 1`, every `i` from 0–6 collapses onto the single X position and the seven labels overlap on the chart; when `data.length === 2`, labels 0 and 6 are at valid positions and 1–5 cluster incorrectly. The Wave 5 parity README confirms this is already visible on Android (`"6mo"` and `"5"` printed verbatim against a single-point trend).

DEFERRED correctly files this to Phase 2.5E, but the fix is a one-liner that prevents a visual bug the founder can screenshot today:

```ts
{data.map((_, i) => (
  <Text key={i} x={toX(i)} … >{xLabelFor(i, data.length)}</Text>
))}
```

…where `xLabelFor` picks from the `labels` array using `data.length` as the slot count. Alternatively: only render labels when `data.length === 7`.

Why it matters: this is the only visible bug that does **not** require the 2.5E range-join to manifest — it's live right now on real data. Fixing here removes one of the items the founder flagged as "first-impression visual bug" from the Wave 5 screenshots.

(Scoping note: if the fix is considered out-of-scope per Rule 8, please at least gate label rendering on `data.length >= 7` so incomplete histories render a bare chart rather than misaligned text.)

#### I-3 · Deep-link handler does not validate `path` shape before `router.push` — accepts any string starting with `"lab-results"`

**File:** `apps/mobile/app/_layout.tsx:56-62`

```ts
if (path.startsWith("lab-results")) {
  if (hydrated) {
    router.push(`/${path}` as never);
  } else {
    pendingLinkRef.current = path;
  }
}
```

`Linking.parse` does not normalise `..` segments, and `path` is trusted as-is. An attacker or fat-finger could send `onlyou://lab-results/../profile` or `onlyou://lab-results/<xss-encoded-garbage>`; Expo Router's matcher will reject unknown routes gracefully, but we're telegraphing to the URL handler that we accept any subpath.

Two small hardenings:

1. Validate the shape: `path === "lab-results" || /^lab-results\/[A-Za-z0-9_-]+$/.test(path)`.
2. Scope the pending-link queue to the same validator so a junk deep-link can't sit in `pendingLinkRef` and fire later.

Why it matters: patient-app deep-links will eventually come from push notifications (Phase 8) and third-party messages. Hardening the accept-list here costs two lines, and the alternative is explaining later why we trusted raw URL input.

### Minor

#### M-1 · `rangePct` has no guard for `low === high`

**File:** `apps/mobile/src/components/biomarker/status-helpers.ts:22-25`

```ts
export function rangePct(v: number, low: number, high: number): number {
  const p = ((v - low) / (high - low)) * 100;
  return Math.max(2, Math.min(98, p));
}
```

When `low === high`, `p` is `±Infinity` or `NaN` depending on `v`. `Math.min`/`Math.max` collapse `NaN` to `NaN`, which then gets stringified into an SVG `x` attribute and silently renders as zero in most runtimes. Every call site today passes range-ful data, but the DEFERRED note for this already flags it. A one-line guard (`if (high === low) return 50;`) would remove the latent failure mode. Already logged in DEFERRED → Wave 5.

#### M-2 · `CategoryFilterPills` horizontal padding `4` is a magic number

**File:** `apps/mobile/src/components/biomarker/CategoryFilterPills.tsx:98`

The ScrollView uses `paddingHorizontal: 4` while the screen-level padding is `24`. DEFERRED notes this for Phase 8 token cleanup; no change needed here.

#### M-3 · `convex/biomarker/patient/myBiomarkerReports.ts` uses `.filter(q.eq(q.field("deletedAt"), undefined))` rather than an index-backed query

**File:** `convex/biomarker/patient/myBiomarkerReports.ts:38,48`

Convex prefers indexed queries over `.filter()` for reads that could touch many rows. At patient scale (< 10 reports per user, < 50 values per report) this is fine today, but if a patient accumulates dozens of soft-deleted reports the linear filter becomes noticeable. No change needed for 2.5D — the projection discipline is more important than the filter optimisation. Worth a note for the 2.5E / Phase 3 performance pass.

#### M-4 · Canonical join within `myBiomarkerReports` is serial (awaits inside a for-loop)

**File:** `convex/biomarker/patient/myBiomarkerReports.ts:44-105`

The inner loop does `await ctx.db.get(...)` (or index query) per value, serially. For a report with 25 values this is 25 round-trips. Convex caches hot reactive queries so the impact in practice is small, but a `Promise.all` around the projection would batch the reads. Not required for 2.5D (patient scale is tiny). Note for later.

#### M-5 · `_layout.tsx` deep-link queue triggers both immediate push and queued push on race

**File:** `apps/mobile/app/_layout.tsx:51-77`

If `Linking.getInitialURL()` resolves after `hydrated` flips true, the URL handler's `hydrated` check will race with the `useEffect` that drains `pendingLinkRef.current`. In practice both paths go through `router.push` and the second push will be a no-op on the same route, but it's worth tightening: clear `pendingLinkRef.current` inside the initial-URL branch once the push fires. Cosmetic.

`process.env.EXPO_PUBLIC_USE_MOCK_BIOMARKERS` is compile-time-stable in Expo, but the expression is re-evaluated on every render. Micro-optimisation: hoist to module scope. No behavioural impact.

#### M-9 · `BiomarkerCard` flat-trend (`value === prev`) always renders honey regardless of `downIsGood`

**File:** `apps/mobile/src/components/biomarker/BiomarkerCard.tsx:119-121`

```ts
const goodTrend = downIsGood ? b.value < b.prev : b.value > b.prev;
const trendColor = goodTrend ? biomarkerPalette.sage : biomarkerPalette.honey;
```

Equality → neither branch true → honey. Since delta is `0%` in the flat case, honey colour feels wrong. A zero-delta case should probably be sage (or muted). Already logged in DEFERRED → Wave 5.

#### M-10 · Two `jest.mock("convex/react", () => ({ useQuery: () => undefined }))` per-file inlines risk masking real-data transform failures

**File:** `apps/mobile/__tests__/screens/lab-results/detail.test.tsx:29-31`, `index.test.tsx:26-28`

The per-file mock forces `useQuery` to return `undefined`, which causes the hook to hit the `data === undefined` in-flight branch — but because `useMock` is `true` under `__DEV__` the hook short-circuits to mocks before that branch ever runs. So today the tests implicitly depend on `__DEV__` being true in jest. If someone later sets `EXPO_PUBLIC_USE_MOCK_BIOMARKERS=0` in the jest env, the tests would continue to pass for the wrong reason (showing a loading state instead of real rows) and `isEmpty` assertions would quietly change meaning. Suggest adding an inline comment above the mock pointing readers at the `__DEV__` short-circuit in `use-biomarker-reports.ts:199-202` so the dependency is documented.

#### M-11 · `react-native-reanimated` mock — `makeEntryAnimation().springify()` silently no-ops

**File:** `apps/mobile/__mocks__/react-native-reanimated.js:14-24`

The chainable factory returns the same `chain` object from every method, which is correct for the current call sites (`FadeInUp.delay().duration().easing()`). If a future author writes `FadeInUp.springify()`, it will compile and test-pass but the real animation config is lost. Consider making the mock a Proxy that warns once on unknown methods, or at minimum add a comment listing the methods the mock currently supports so the failure mode is discoverable.
