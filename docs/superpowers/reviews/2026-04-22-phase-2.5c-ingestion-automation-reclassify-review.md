### IMPORTANT

#### [I-1] `admin.generateUploadUrl` and `admin.getE2EStatus` are public `mutation`/`query` with no caller-identity check

**Files:**

- `convex/admin.ts:90-96` — `generateUploadUrl = mutation({ args: {}, handler: assertNotProd + ctx.storage.generateUploadUrl() })`
- `convex/admin.ts:105-136` — `getE2EStatus = query({ args: { labReportId }, handler: assertNotProd + returns full labReport + biomarkerReport + values + notifications for the owning user })`

**What's wrong:**

Both are exported as public `mutation`/`query` (not `internalMutation`/`internalQuery`) so they are reachable by any caller with the Convex deployment URL and no auth credentials. The only guard is `assertNotProd()`, which fails closed on production but is a no-op on `dev-deploy`. On dev, an attacker with the public Convex URL can:

1. Mint unlimited Convex storage upload URLs (`generateUploadUrl` → `ctx.storage.generateUploadUrl()`), each usable for arbitrary file uploads.
2. Read arbitrary patients' `labReport + biomarkerReport + biomarker_values + notifications` given only a `labReportId` (which is guessable across the sparse dev keyspace).

The code comments call this out honestly (`"ConvexHttpClient cannot call internal mutations from outside Convex; assertNotProd() is the real gate"`) and the target use case (the local `run-manual-e2e.ts` driver) is legitimate. But "dev only" isn't the same as "safe on dev." A dev deployment URL leaks frequently during reviews / screenshares / error messages. This is the first public read endpoint in the codebase that returns unrestricted user data.

**Why it matters:**

- The `retryParseLabReport` mutation spent a whole Wave 1 review cycle (C-1 in that review) getting session-token auth because the rule is "if it reads or writes user data, it authenticates." These two endpoints bypass that rule because they're labeled "dev-only," but they're public and carry the same data.
- The pattern sets precedent. Phase 5 admin portal will add 10+ admin-role actions — if the first rough draft lands without identity checks because "assertNotProd is the real gate," the precedent bites.

**How to fix (pick one):**

1. **Make them `internalMutation` / `internalQuery` and drive the script via a Convex CLI (`npx convex run admin:generateUploadUrl '{}'`).** This matches `seedAdminUser` (already internal) and removes the public surface entirely. The E2E script invokes via `spawnSync("npx convex run ...")` the same way `seed-admin-user.ts` does.
2. **Require a dev shared-secret token.** Add `args: { devToken: v.string() }`, compare against `process.env.E2E_DEV_TOKEN` in the handler, throw if mismatch. The E2E script already reads env; add one more var.
3. **Accept the risk, file a DEFERRED ledger entry naming Phase 5 (admin portal) as the destination for replacing these with real admin-role–authenticated surfaces, and add a deployment-level allowlist doc.** Minimum effort, paper trail preserved.

Either way: ship a decision note and a DEFERRED entry if option 3. The current state ships two functional holes on dev that aren't documented anywhere except inline code comments.

#### [I-2] `simulateLabUpload` bypasses `assertPortalEnabled("LAB")` even though it emulates the LAB portal upload path

**File:** `convex/admin.ts:48-78`

**What's wrong:**

`simulateLabUpload` calls `createLabReportFromAction` with `source: "lab_upload"` — the same source the real `labUploadResult` mutation uses. The real mutation is gated by `assertPortalEnabled("LAB", deployment)`, which requires both `LAB_PORTAL_ENABLED=1` and (on prod) `LAB_PORTAL_REAL_AUTH=1`. The admin simulator has neither gate, only `assertNotProd()`.

The spec's intent for the double-flag pattern (`docs/superpowers/specs/...-design.md` Wave 5 section) is "LAB portal endpoints can never run on prod without explicit real-auth opt-in." By providing an admin action that produces the exact same effect (`lab_orders.status → results_uploaded` + `lab_report_uploaded_for_you` notification… wait, no — `simulateLabUpload` uses `createLabReportFromAction` which patches `lab_orders.status → results_uploaded` but does NOT emit the notification since that's only done in the mutation-path. So the kill-switch semantics differ subtly.)

This is less severe than I-1 because `assertNotProd()` fails closed on prod and the simulator is an action (dashboard-only, not public-facing via HTTP). But it introduces a bypass path around the portal gate that the plan explicitly built, and the two paths have subtly different side effects.

**Why it matters:**

- Asymmetry between the "real" lab-portal path and the "dev-stub" admin path means a Phase 4+ change to `assertPortalEnabled` semantics won't apply to admin-triggered simulations. That's the exact class of bug the shared `assertPortalEnabled` helper was meant to prevent.
- A dev-deploy operator disabling the lab portal via `LAB_PORTAL_ENABLED=0` (e.g., to block a partner until onboarding re-opens) would still see admin-triggered simulations go through — because `simulateLabUpload` doesn't honor that flag.

**How to fix:**

Add `assertPortalEnabled("LAB", process.env.CONVEX_DEPLOYMENT ?? "")` right after `assertNotProd()` in `simulateLabUpload`. One line. The existing `LAB_PORTAL_ENABLED` env check is already what the dev operator expects to flip — making the admin simulator honor the same flag is a uniform invariant.

If the intent is "admin simulations must work even when the LAB portal is disabled for partner misuse," that's a defensible choice — but call it out in a comment and/or decision note, and emit a different `source` (e.g., `"admin_simulation"`) so the resulting `lab_reports` row is distinguishable from real partner uploads. As it stands, real + simulated rows look identical in the DB.

#### [M-2] `simulateLabUpload` diverges from the "partner notification" side effect of `labUploadResult`

**Files:**

- `convex/biomarker/portal/labUploadResult.ts:46-49` — emits `lab_report_uploaded_for_you` notification for `order.userId`
- `convex/admin.ts:68-77` — `createLabReportFromAction` only; no notification emit

**What's wrong:**

A real lab upload notifies the patient immediately (`"Your lab uploaded a report — we're analyzing it"`). An admin-simulated upload doesn't, so the dev E2E flow doesn't exercise that notification's emit path. The `run-manual-e2e.ts` script polls `getE2EStatus` which returns `notifications`, but will only ever see `lab_report_ready` / `lab_report_parse_failed` from the parseLabReport action path — never `lab_report_uploaded_for_you`.

Either emit the notification from `simulateLabUpload` (one call to `writeNotificationFromAction`) or document that the E2E flow deliberately skips that notification to exercise the patient-upload path, not the lab-upload path. Since the E2E script uses `source: "lab_upload"`, the former is the right fix.

#### [M-4] Prod-regex is duplicated across three files; drift risk reintroduced even after the Wave 5 fix

**Files:**

- `convex/lib/envGuards.ts:8-10` — canonical `PROD_DEPLOYMENT_PATTERNS: [/(^|-)(prod|production)(-|$)/i]`
- `scripts/seed-admin-user.ts:16` — same regex inlined "Must match convex/lib/envGuards.ts"
- `scripts/run-manual-e2e.ts:22` — same regex inlined "Must mirror convex/lib/envGuards.ts"

The Wave 5 review's I-1 finding was resolved inside `convex/` by hoisting `PROD_DEPLOYMENT_PATTERNS` into `envGuards.ts` and importing from both `admin.ts` and `portalGates.ts`. Wave 6 added two new `scripts/*.ts` files that couldn't easily import from `convex/` (different tsconfig, no Convex codegen dependency) and pragmatically inlined the regex with a "must match" comment.

The "must match" comment is honest, but three copies of one regex is the exact pattern the Wave 5 review warned would regress (review §I-1 final paragraph). Options:

1. Hoist the regex to `packages/core/src/env/prodDeploymentPattern.ts`, export from `@onlyou/core/env`, import in all three call sites. `@onlyou/core` is already part of the monorepo; adding one export is trivial.
2. Add a tiny test that reads all three files and asserts the regex literal is string-identical (cheap, effective).

Option 1 is the right long-term fix. Option 2 is a 5-minute bandaid. At a minimum, a `CONSISTENCY: grep for /(^|-)(prod|production)(-|$)/i across the repo before editing` note in CLAUDE.md or the phase-review template would prevent re-litigation.

#### [M-5] `writeAuditLog` silent-skip on missing sentinel admin is correct behavior but not covered by a negative test

**File:** `convex/biomarker/internalMutations.ts:467-489`

When `adminUserId === null` and no user has `role: "ADMIN"`, `writeAuditLog` returns `{ skipped: true as const }` and writes nothing. The callers (`reclassifyAllReports`, `reclassifyForCanonicalId`) ignore the return value. The `reclassify-roundtrip.test.ts` explicitly seeds an ADMIN sentinel to prevent this silent-skip during the happy-path test.

But no test asserts the silent-skip actually happens when no admin exists — a developer deleting the sentinel seed line from `reclassify-roundtrip.test.ts` would see the test still pass because the commit returns and notifications still emit; only the audit row would be missing, which the test doesn't check. And no test covers "action runs end-to-end without a seeded ADMIN and emits observable behavior on the skipped path (e.g., a telemetry event)." Today the skip is invisible.

Add a short test: seed no ADMIN user, call `reclassifyAllReports` in commit mode, assert `admin_audit_log` is empty AND the action still completes AND notifications still emit. This locks in the "best-effort audit in dev" contract documented in the handler comment.
