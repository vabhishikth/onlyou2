## Summary

A disciplined, scope-honest implementation of the 2.5B parse pipeline. The nine-class error taxonomy is mostly correctly wired, the pregnancy guard runs before the profile guard (the single most load-bearing ordering invariant in the whole phase), idempotency by `contentHash`/`biomarker_reports` short-circuit is clean, the retry scheduler is a pure function with per-class schedules exactly matching spec §2.3, and the cache breakpoint landed on the system block per the mid-execution decision. The 1128-line branch-coverage test suite genuinely exercises the branches it claims to, including intentional documentation of two code-vs-spec mismatches (test `parseLabReport.branches.test.ts:589` and `:1059` openly note "this is the actual behavior, not the plan"). That honesty is good; the underlying behaviors are the findings below.

**Green lights:**

- **Pregnancy guard runs first — with the correct subtlety.** `classifyRow.ts:107-118` finds the matching range **before** the pregnancy check, then uses `matchingRange?.pregnancySensitive` to gate. This is the right implementation: checking `pregnancyStatus === "unknown"` alone would paper every pregnancy-unknown female's report with unclassified rows; checking it only when the specific marker is pregnancy-sensitive is the intended narrow escape hatch. The ordering (range lookup → pregnancy → profile → qualitative → unit) is genuinely what the spec §3.5 requires.
- **Error taxonomy mostly matches §2.2.** 7 of 9 classes land in the right bucket: zod→terminal, max_tokens→in-process retry→terminal, refused→one reprompt→terminal, is_lab_report:false→`not_a_lab_report`, pdf_decode→terminal, 429→cron with retry-after honored, 5xx/network→cron per §2.3 schedule. `max_retries_exceeded` is correctly computed from both wall-clock cap and slot exhaustion (`retryScheduler.ts:30-47`). (The 400 case is a Critical finding — see below.)
- **Retry scheduler is a pure function with named constants.** `WALL_CLOCK_CAP_MS = 30*60*1000`, `NETWORK_5XX_SCHEDULE_MS = [30_000, 120_000, 300_000, 900_000]` — the 30s → 2m → 5m → 15m ladder from spec §2.3 is a literal array, trivially auditable. Wall-clock cap is checked before slot computation, so a `retryCount: 2, firstAttemptAt: 31min ago` request terminates as `max_retries_exceeded` rather than trying slot 2. Slot exhaustion (`attempt - 1 >= schedule.length`) is a second, independent terminal path. Both are individually tested.
- **Idempotency is enforced at two layers.** `parseLabReport.ts:62-68` checks for an existing `biomarker_reports` row by `labReportId` and short-circuits with `outcome: "idempotent_noop"` before marking analyzing or touching Claude — so a re-trigger after success is free. The curation-queue upsert uses a `normalizedKey` (normalized name + unit) with unique index `by_normalized_key` so the same unknown marker across two reports increments `occurrenceCount` rather than inserting. Tested in scenarios 15 and 5 respectively.
- **Cache breakpoint on system block is correct per the mid-execution decision.** `convex/lib/claude.ts:160-169` puts `cache_control: { type: "ephemeral", ttl: "1h" }` on the single system block, with `BETA_HEADER_EXTENDED_CACHE = "extended-cache-ttl-2025-04-11"` in the `betas` array. This matches `docs/decisions/2026-04-18-cache-breakpoint-on-system-block.md` and is robust to the `FEW_SHOT_EXAMPLES` array being empty (which it currently is — see Important I-3). The comment on lines 116-118 explicitly documents why this placement guarantees exactly one cache_control marker regardless of few-shot count — that's the right design rationale to record in code.
- **Telemetry never logs patient data.** `convex/lib/telemetry.ts` is a single 76-line module with a tight `ParseLogFields` type that restricts callers to `{ labReportId, hashedUserId, event, modelId, retryCount, durationMs, markerCount, errorCode, contentHashPrefix, alert }` — no marker names, no values, no patient name, no raw PDF text. `userId` is the only potentially-sensitive field and it is always replaced with its FNV-1a hash before emission (`logParseEvent:62-66`). Salted with a stable project-wide constant so hashes are comparable across log lines. The pure-JS FNV-1a is justified by `docs/decisions/2026-04-18-telemetry-hash-pure-js.md` (Convex V8 runtime has no `node:crypto`). 12-hex-char width is enough collision-resistance for a grep-tag; not cryptographic, but also not claimed to be.

Small but correct — avoids a guaranteed API round-trip for non-lab-reports that somehow slipped past the `is_lab_report: false` short-circuit.

## Findings

### Critical

**C-1 — Anthropic 400 does not terminate per spec §2.2; it is routed through `scheduleRetry` and treated as 5xx/network.**

File: `D:/onlyou2-phase-2.5b/convex/biomarker/parseLabReport.ts:358-374`

```ts
function classifyError(err: unknown): RetryClass {
  const status = (err as { status?: number }).status;
  if (status === 429) { ... return { kind: "rate_limit", ... }; }
  if (status === 400) {
    // Not retryable — surfaced by caller as terminal; shouldn't reach here
    return { kind: "network_or_5xx" };
  }
  return { kind: "network_or_5xx" };
}
```

The comment claims "shouldn't reach here," but nothing upstream actually intercepts 400 errors and routes them to `terminalFail`. `callExtraction` lets the SDK throw on 400 (standard Anthropic client behavior), `extractMarkersWithRetry` only catches `max_tokens` and `ExtractionError`, so a real 400 from the API bubbles all the way to the orchestrator's `catch` block on line 116, which sees a non-`ExtractionError` and routes to `scheduleRetry(…)`. `scheduleRetry` calls `classifyError`, which falls into the `status === 400` branch and returns `{ kind: "network_or_5xx" }` — from there, the row is marked for a 30-second cron retry, exactly the opposite of the spec's "terminal + alert: p1 log" requirement.

The existing test at `parseLabReport.branches.test.ts:567-592` makes this bug explicit: the assertion is `expect(["failed:api_bad_request", "retry_scheduled"]).toContain(result.outcome)` with a long prose comment saying "we test the actual behavior, not the plan." Accepting both outcomes is not testing; it's documenting a known defect. The spec says 400 is terminal with `errorCode: "api_bad_request"` and an ERROR-level `alert: "p1"` log — neither happens currently.

Why this matters: 400 bad-request means our request is structurally wrong (malformed PDF, bad API schema, model ID typo). Retrying an identical request 4 times over 22 minutes wastes tokens, delays the terminal state that signals "the pipeline is broken," and — crucially — prevents the `alert: "p1"` signal from surfacing at all (the p1 branch in `terminalFail` is never reached since `scheduleRetry` is called instead). This is the one alert path that exists before Phase 8's hosted pager, and it's silently unreachable.

**Recommended fix** — add an early terminal branch in the orchestrator's `catch`:

```ts
} catch (err) {
  if (err instanceof ExtractionError) {
    return await terminalFail(ctx, labReportId, labReport, err.errorCode, now);
  }
  // 400 is structurally terminal — never retry
  if ((err as { status?: number }).status === 400) {
    return await terminalFail(ctx, labReportId, labReport, "api_bad_request", now);
  }
  return await scheduleRetry(ctx, labReportId, labReport, err, now);
}
```

Then tighten the existing test to assert a single outcome (`failed:api_bad_request`) and add a new assertion that `logParseEvent` was called with `level: "error"` and `alert: "p1"`.

### Important

**I-1 — Retry cron can double-fire a row between scheduling and the scheduled action acquiring `lockedAt`.**

File: `D:/onlyou2-phase-2.5b/convex/crons.ts:22-30`
File: `D:/onlyou2-phase-2.5b/convex/biomarker/internalMutations.ts:5-16` (`markAnalyzing`)

The cron queries `findRetryCandidates` (status=analyzing, nextRetryAt ≤ now, lockedAt null or ≥ 90s old) and schedules `parseLabReport` via `ctx.scheduler.runAfter(0, …)` — but does **not** claim the lock itself. `lockedAt` is set later, inside `parseLabReport`'s `markAnalyzing` call (line 71). Between the cron's scheduling and the scheduled action starting, the row still has `status: analyzing`, `nextRetryAt: <past>`, `lockedAt: null` (or still stale). If the next cron tick fires before the scheduled `parseLabReport` has started executing, it will re-pick the same row.

The Convex scheduler typically runs scheduled-at-0 actions within milliseconds, but under load (e.g., a burst where `findRetryCandidates` returns 25 rows) or in the stale-lock release case (where `parseLabReport` is crashing mid-way and dropping the lock), the cron's 2-minute cadence means one missed action could be scheduled twice. Spec §2.4 explicitly names "cron cannot double-fire" as a design invariant.

**Recommended fix** — claim the lock in the cron, inside the same mutation that finds candidates:

```ts
// internalMutations.ts — new mutation
export const claimForRetry = internalMutation({
  args: { now: v.number(), staleLockCutoff: v.number() },
  handler: async (ctx, { now, staleLockCutoff }) => {
    const candidates = await ctx.db
      .query("lab_reports")
      .withIndex("by_next_retry", (q) => q.lte("nextRetryAt", now))
      .filter((q) =>
        q.and(
          q.eq(q.field("status"), "analyzing"),
          q.or(
            q.eq(q.field("lockedAt"), undefined),
            q.lt(q.field("lockedAt"), staleLockCutoff),
          ),
        ),
      )
      .take(25);
    const claimed: Id<"lab_reports">[] = [];
    for (const row of candidates) {
      await ctx.db.patch(row._id, { lockedAt: now });
      claimed.push(row._id);
    }
    return claimed;
  },
});
```

Then the cron calls `claimForRetry` (a mutation, atomic per row) before `scheduler.runAfter`, and `parseLabReport` no longer re-stamps `lockedAt` on entry (or, if it does, that's an additional fine-grained lock for the parse itself — still safe because the cron's claim is already in place). This also makes the 90s stale-lock timeout meaningful end-to-end: a crashed cron-scheduled parse will have `lockedAt` set by the cron, and the next cron only picks it up after the 90s release window.

The existing scenario 16 test doesn't catch this because it only exercises `findRetryCandidates` as a pure query, not the cron-to-action handoff. A double-fire test would race two simultaneous cron invocations against one stuck row and assert `parseLabReport` ran exactly once.

**I-2 — Refusal-detection false-positive risk on valid reports containing clinical text.**

File: `D:/onlyou2-phase-2.5b/convex/biomarker/internal/extractMarkers.ts:59-69`

```ts
const REFUSAL_PATTERNS = [
  /can'?t help/i,
  /medical advice/i,
  /i['']m not able/i,
  /not trained/i,
];

function looksRefused(raw: unknown): boolean {
  const text = JSON.stringify(raw);
  return REFUSAL_PATTERNS.some((p) => p.test(text));
}
```

`looksRefused` stringifies the entire response and regex-matches — which means a lab report's `lab_printed_range` field saying "Consult physician for medical advice" would falsely trigger the refusal path and burn a reprompt call. The patterns also match on arbitrary substrings (`/medical advice/i` would match "no medical advice required" or "medical advice line: 1800-..."). Since `patient_name_on_report` and `name_on_report` are also included in the stringified blob, any report from a lab that puts "Medical Advice: see your doctor" in its footer also hits this.

Low-probability on synthetic fixtures (which don't have such boilerplate), but the layout-clone private fixtures and all real reports will. The impact is bounded — one extra API call + guaranteed-valid second response — but still a correctness concern because `patient_name_on_report: "I can't help with medical advice"` in scenario 10 (the refused-twice test) is explicitly constructed to trigger this pattern. The pattern is the assertion, not the API contract.

**Recommended fix** — only check the top-level string fields Claude would produce on refusal (typically a single text block with no structured JSON at all), or switch to checking the presence of an invalid-shape response first:

```ts
function looksRefused(raw: unknown): boolean {
  // Valid-shape responses (even with clinical text) are not refusals.
  if (typeof raw === "object" && raw !== null && "markers" in raw) return false;
  const text = typeof raw === "string" ? raw : JSON.stringify(raw);
  return REFUSAL_PATTERNS.some((p) => p.test(text));
}
```

Or better: gate on `typeof raw.content[0].text === "string" && raw.content[0].text.startsWith("I")` — refusals are conversational prose, not JSON objects. Either way, avoid blind stringification.

**I-3 — `FEW_SHOT_EXAMPLES` is empty; spec §2.5 calls for 8. Cost model and behavior stability both depend on this.**

**Recommended fix** — one of:

1. Populate the 8 few-shot examples before merge. The spec names them (§2.5 table); the 3 golden extractions already committed under `fixtures/golden/*.json` cover 3 of the 8.
2. Explicitly defer to 2.5C with a ledger entry in `docs/DEFERRED.md` and update spec §2.5 to reflect the new reality. "Populated by Task 10" is not a ledger entry.

Option 1 is the correct answer under the CLAUDE.md rule 9 ("every entry names a destination phase; 'later' is not valid"). The comment "Populated by Task 10" is dead — Task 10 has already shipped without the content.

### Minor

**M-1 — `extractMarkers.ts:153-158` silently converts network-error-during-zod-retry into `zod_validation` terminal.**

File: `D:/onlyou2-phase-2.5b/convex/biomarker/internal/extractMarkers.ts:140-160`

The zod-retry `try/catch` on line 140 re-throws `ExtractionError` as-is but wraps **any other error** (network 503, 429, DNS) as `new ExtractionError("zod_validation", ...)`. Net effect: if the first parse fails zod validation and the "return only JSON" follow-up call hits a 503, the 503 is converted to a terminal zod error rather than a retryable network error. Scenario at `parseLabReport.branches.test.ts:1044-1061` documents this with "this is the actual behavior" prose, which (like C-1) is honesty, not correctness.

The impact is narrower than C-1 because the zod-retry path is rare and the 30-min wall-clock cap still applies — a user whose first parse hit zod fail + whose second parse hit a network blip just gets told "format error, re-upload" instead of "please wait, retrying." Annoying but not data-losing. Fix: re-throw non-`ExtractionError` from the zod-retry catch block so the orchestrator's `scheduleRetry` can see it. Current behavior is test-pinned, so fix + test update go together.

## Assessment

**APPROVE-WITH-FIXES.** The phase delivers on its scope — a working end-to-end parse pipeline with real Claude Vision calls, 9-class error taxonomy mostly wired, pregnancy-first classification, retry cron with explicit locking, idempotency by `contentHash`/existing-report, and a live-integration suite that CI-gates on the fixtures. The architecture is sound; the critical fixes are all in one-to-three-line changes to already-structured code, not design-level rework.

The one Critical finding (400 handling) is narrow in scope but materially defeats the spec's one surviving alert mechanism. The Important findings are real but bounded. Fix them, update the two "actual behavior not plan" tests, and merge. If I-3 (few-shots) is deferred rather than filled, that's acceptable **only** with a `docs/DEFERRED.md` entry naming a destination phase and the spec §2.5 updated accordingly — "later" is not allowed per CLAUDE.md rule 9.

Strong execution overall. The willingness to write a test that documents "this doesn't match the spec yet" is healthier than quietly changing the assertion to match broken code — but those tests are a to-do list, not a completed review. Convert them before merge.
