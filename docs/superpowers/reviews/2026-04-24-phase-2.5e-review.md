### I-2 · Real-data explainer lookup always falls back

**File:** `apps/mobile/src/hooks/use-biomarker-reports.ts:196` → `apps/mobile/app/lab-results/[id].tsx:137`
**Problem:** The hook maps `id: v._id` (Convex document id) onto the `BiomarkerMock.id` field, then Detail calls `explainerFor(b.id, b.name, b.status)`. `EXPLAINER_MAP` is keyed by mock ids (`"ldl"`, `"hdl"`, etc.) and canonical strings — never doc ids. Every real-data biomarker detail will render the generic fallback ("This marker is outside our reference database…"), silently undoing one of the headline Phase 2.5E deliverables when `EXPCO_PUBLIC_USE_MOCK_BIOMARKERS=0`.

The hook header comment acknowledges this ("real Convex data currently uses doc IDs … will hit the fallback until canonical-id threading lands in a later phase"), but it is not captured in `docs/DEFERRED.md` with a destination phase, and it directly contradicts the phase goal of "status-aware explainer copy, threaded through the mobile UI."

**Recommended fix:** Either (a) map `id: v.canonicalId ?? v._id` in `transform()` (one-line change — canonical ids already key the explainer map), or (b) thread a dedicated `explainerKey: v.canonicalId` onto `BiomarkerMock` and have Detail prefer it over `id`. Add a DEFERRED entry if (b) is chosen and punted.

### I-3 · `collectionDate` ISO-8601 contract is not enforced at ingest

**File:** `convex/schema.ts:158,182,201` + `convex/biomarker/parseLabReport.ts:204-205,361`
**Problem:** The trend sort in `myBiomarkerReports.ts:134-137` assumes `collectionDate` is ISO-8601 so `localeCompare` yields chronological order. The schema only validates `v.optional(v.string())`. `parseLabReport` pipes `extract.response.collection_date` (AI-produced) straight into the row with no format validation. A non-ISO string ("10/04/26", "April 2026") would silently produce an incorrect trend ordering and wrong `prev`. Tests pass only because they use same-length numeric strings from `String(Date.now())`, which happen to sort numerically.

**Recommended fix:** Add a lightweight ISO-8601 guard (regex `^\d{4}-\d{2}-\d{2}`) at the `internal.biomarker.internalMutations.createBiomarkerReport` / `insertBiomarkerValue` boundary. Reject or normalise at the door. Backstop the query with a second sort key (e.g. classifiedAt) to make the sort total-order-stable if the string contract degrades.

## Minor findings

### M-1 · Redundant trend lookup across reports in the same query

**File:** `convex/biomarker/patient/myBiomarkerReports.ts:55-166`
**Problem:** The outer `for (const report of reports)` loop processes every report the patient owns, and the inner `Promise.all` fans out a full `by_user_canonical_date` history lookup for every value on every report. For N reports × M markers, that is N×M history reads, even though the UI only consumes the most-recent report (hook `transform()` drops all others). For a patient with 5 reports × 20 markers, that is ~100 redundant scans.

**Recommended fix:** Either (a) compute the trend once per canonicalId across all the patient's values (group before fan-out), cache the result, and reuse across reports; or (b) short-circuit the hook/query contract to "latest report only" if the UI genuinely never renders older reports. Option (b) is smaller. Not required at current patient scale — flag as perf debt with a DEFERRED entry.

### M-2 · Seed `assignStatus` duplicates `classifyRow.assignStatus`

**File:** `convex/seed/devBiomarkerReport.ts:216-225` ⇄ `convex/biomarker/internal/classifyRow.ts:216-229`
**Problem:** The two implementations are currently identical (`<=`/`>=` + strict `<`/`>` ordering) but there is no compile-time link. If `classifyRow.assignStatus` is ever tweaked (e.g. the threshold-invariant decision is revisited), the seed will drift and produce differently-classified data than the real pipeline, which is exactly the scenario live-E2E is meant to catch.

**Recommended fix:** Export `assignStatus` (and the minimal `ReferenceRange` subset it needs) from `classifyRow.ts` and import it into `seedOneReport`. One-line extraction + one-line import. Pure dedup.

### M-3 · `rangeDirection` synthesis conflates null vs absent actionable bound

**File:** `apps/mobile/src/hooks/use-biomarker-reports.ts:184-189`
**Problem:** `rangeDirection` is derived from whether `actionBelow`/`actionAbove` are null, but the fallback branch (`r = null`) returns `"bidirectional"` implicitly. For a fully-unclassified marker that has no range at all, this still produces a bidirectional RangeBar gradient with synthesised `optLow`/`optHigh` = 25/75 — visually fine but semantically it is "no range known," not "bidirectional." Low impact because such values are filtered by `if (!v.canonical && v.status === "unclassified") continue;` just above. Worth a code comment calling out the invariant.

**Recommended fix:** Add an explanatory comment at line 184 noting that `r === null` only reaches here when the marker has `canonical` but no range row (rare), and that bidirectional is a reasonable visual default in that case.

### M-4 · `EXPLAINER_MAP` has duplicate entries for canonical/mock key pairs

**File:** `apps/mobile/src/data/biomarker-explainers.ts` (e.g. `plt`/`platelets`, `b12`/`cobalamin`?)
**Problem:** 31 claimed canonical entries, but some are mock-key aliases of others. Any future copy tweak needs to be mirrored, easy to miss.

**Recommended fix:** Either factor aliases into a `const aliases = { platelets: 'plt' }` indirection, or add a test asserting that alias pairs stay in sync. Low urgency — the test coverage on `explainers.test.ts:19` already enforces shape, not content equality.

### M-5 · `__DEV__` guard for mock in hook assumes React Native globals

**File:** `apps/mobile/src/hooks/use-biomarker-reports.ts:227-229`
**Problem:** `typeof __DEV__ !== "undefined" && __DEV__` is safe, but unit tests that import `transformForTest` or this hook outside of a Metro bundle will see `__DEV__` undefined → no mock path → fine. Only worth noting because a future Node-based integration test might trip on this if it calls `useBiomarkerReports()` directly. Non-blocking.

**Recommended fix:** None required. If Node-side testing becomes necessary, extract the mock-toggle into a pure helper and inject it.

## What was done well

- **Test discipline.** The `myBiomarkerReports.test.ts` suite (477 new lines) locks range projection, prev-null, soft-delete exclusion, ISO-8601 ordering, and unclassified range=null — exactly the invariants the hook transform depends on. `use-biomarker-reports.test.ts` (186 lines) covers direction-aware status mapping, unbounded range synthesis, and latest-report semantics. This is the right granularity.
- **Index choice is deliberate.** The M-3 decision to keep soft-delete as a `.filter()` (documented inline at line 20 and in a decisions note) rather than chasing a by_deleted index shows correct hot-path reasoning.
- **RangeBar gradient null-filtering.** The `.filter((node): node is React.ReactElement => node !== null)` at line 189 avoids a known `react-native-svg` crash on nullable children. Good defensive polish with a type-narrowing predicate — no `any`.
- **IST greeting helpers are pure.** `getIstGreeting` / `getIstDateLine` use `Intl.DateTimeFormat` with a fixed timezone, no Date mutation — cleanly unit-testable, locale-stable.
- **Seed history math is legible.** The `(reports - 1 - i)` drift formula with oldest-first iteration is both correct and easy to reason about. The per-report unique `fileId` via fresh `ctx.storage.store` avoids contentHash collisions.
- **No hardcoded hex.** All RangeBar colours route through `biomarkerPalette` tokens — per CLAUDE.md rule.
- **Session-token auth pattern honoured.** No `ctx.auth.getUserIdentity()` anywhere in the diff; all auth goes through the `sessions` table lookup with expiry check.
