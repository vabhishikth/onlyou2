## Issues

### Critical (Must Fix)

#### C-1: `biomarkerReportsForPatient` + `simulateLabUpload` use `ctx.auth.getUserIdentity()` but the codebase has **zero Convex identity providers wired**

**Files:**

- `convex/biomarker/portal/biomarkerReportsForPatient.ts:14‚Äì20`
- `convex/admin.ts:62‚Äì63`
- Decision on record: `docs/decisions/2026-04-20-session-token-auth-in-2-5c-mutations.md`
- Supporting evidence: `convex/auth.config.ts` ‚Äî `{ providers: [] }`

**What's wrong:**

The 2026-04-20 decision note is explicit: \_"ONLYOU v2 never wired Convex's built-in identity (no `auth.config.ts`, no `ConvexProviderWithAuth`)‚Ä¶ **All Phase 2.5C mutations that need auth accept a `token: v.string()` arg and resolve via the session-token pattern used by `users.ts`. This matches the spec intent and every other mutation in the codebase."** That decision was written precisely because the earlier `intakeUpload` commit had the same mistake and had to be reverted (see its code-review C-1 on commit `2d10125`).

Wave 5 reintroduces the exact same pattern the decision note ruled against. `convex/auth.config.ts` has an empty `providers: []` array ‚Äî in any real deployment (dev-deploy included), `ctx.auth.getUserIdentity()` will return `null` every time, and both endpoints will throw `unauthenticated` on every call.

The tests pass only because `convex-test`'s `t.withIdentity({ subject, issuer })` bypasses the provider config ‚Äî a local-harness-only escape hatch. There is no production path that reaches the happy branch.

**Why it matters:**

- `biomarkerReportsForPatient` is **completely non-functional** on any real deployment the doctor portal could call. Phase 4 will find this on first wire-up.
- `simulateLabUpload` will **never run** in the Convex dashboard (the intended smoke-test path per the plan Step 2) because dashboard calls don't provide a Convex identity either. This likely explains why the commit message says "No dashboard smoke yet (requires convex auth)" ‚Äî the code was written assuming Phase 4 wires auth, but nothing in 2.5C makes that happen.
- Violates Rule 7 ("keep it simple") and the explicit precedent set by the decision note.
- The other 2.5C mutation that needs auth (`retryParseLabReport.ts:14‚Äì17`) has a code comment calling out this exact point: _"Plan text prescribed getUserIdentity(); this file overrides that per the decision note to match the codebase."_ Wave 5 didn't follow suit.

**How to fix:**

Two options, pick one:

1. **Preferred ‚Äî match the codebase.** Add `token: v.string()` to both functions' args. Resolve the caller via `sessions.by_token`. Mirror `retryParseLabReport.ts:27‚Äì44`. Update tests to seed a `sessions` row and pass `token` instead of using `withIdentity`. This is what the decision note prescribed.

2. **If the intention really is "doctor portal will bring its own auth in Phase 4"** ‚Äî add a DEFERRED ledger entry that says so, add a failing smoke-test harness that documents the gap, and wrap both functions in an `assertNotProd()`-style guard so the current "both endpoints are broken" state doesn't silently ship to any env where someone might call them. Then file a follow-up ticket. But this is strictly worse than option 1 because it leaves two dead endpoints in the registry.

Either way: the current state is "ships code that cannot run end-to-end on any non-test deployment." This is the merge-blocker.

**What's wrong:**

**Why it matters:**

- Asymmetry between `assertNotProd` and `assertPortalEnabled` means two different "is this prod?" definitions coexist in the same `convex/` tree. A future maintainer doing string-search-driven review will miss one.
- The 2.5A review already litigated this and settled on the anchored form. Wave 5 regressed the decision.
- Low risk today (deployment is `dev-deploy`), but the wrong pattern to commit.

**How to fix:**

Either:

- Share the same patterns. Export `PROD_DEPLOYMENT_PATTERNS` from `admin.ts` (or better, hoist it to `convex/biomarker/lib/portalGates.ts` or a new `convex/lib/envGuards.ts`) and use it in both places.
- Or at minimum tighten to `/^(prod|production)$|-(prod|production)$/i` and update the test case that currently passes with `"prod-deploy"` (which also matches the tighter form, so the test stays green).

The unit test at `portal-contracts.test.ts:19` should grow a case like `"dev-reproductive-care"` or `"prodigy-test"` that asserts the loose form's false-positive ‚Äî then failing that test forces the tightening.

#### I-2: `biomarkerReportsForPatient` returns raw docs including `userId`, `deletedAt`, `_creationTime` ‚Äî PHI surface shape is not audited

**File:** `convex/biomarker/portal/biomarkerReportsForPatient.ts:28‚Äì37`

**What's wrong:**

The query pushes `{ report, values }` straight from `ctx.db.get` / `.collect()` into the response. That means the wire payload includes:

- `userId` (patient Convex ID)
- `deletedAt` (always `undefined` here because we filtered on it, but it's in the schema)
- `_creationTime` (internal Convex metadata)
- `updatedAt` on values (if the field exists)
- Any future field added to `biomarker_reports` or `biomarker_values` becomes automatically doctor-visible with zero review.

**Why it matters:**

- The doctor portal is the **first external-party read surface** for biomarker data in the codebase. This is the moment to decide what the wire contract is, not after Phase 4 starts building UI against it.
- The implicit "whatever is in the row" contract means schema migrations silently change the portal response shape. No type test, no snapshot. Wave 4 adds reclassify audit fields to this same table ‚Äî if those fields aren't meant to be doctor-facing, they will be.
- `ONLYOU-SOURCE-OF-TRUTH.md` and `DESIGN.md` emphasize that lab/pharmacy/delivery "never see diagnosis." The doctor portal is allowed to, but the wire contract should still be deliberate.

**How to fix:**

Add an explicit projection. Even a minimal one:

```ts
return reports.map((report) => ({
  report: {
    _id: report._id,
    collectionDate: report.collectionDate,
    narrative: report.narrative,
    counts: {
      optimal: report.optimalCount,
      subOptimal: report.subOptimalCount,
      actionRequired: report.actionRequiredCount,
      unclassified: report.unclassifiedCount,
    },
    analyzedAt: report.analyzedAt,
    lastReclassifiedAt: report.lastReclassifiedAt,
  },
  values: values.map((v) => ({
    /* explicit fields */
  })),
}));
```

If explicit projection is too much scope for a stub, add a **DEFERRED ledger entry** naming the destination phase (likely Phase 4) and a TODO comment on the return statement so Phase 4 doesn't paint the wire contract through by accident.

#### I-3: `labUploadResult` doesn't validate `fileSizeBytes >= 0` nor cap before delegating

**File:** `convex/biomarker/portal/labUploadResult.ts:20‚Äì44`

**What's wrong:**

The mutation trusts the caller's `fileSizeBytes` and passes it straight to `createLabReportFromMutation`. The shared helper does call `validateSizeAndMime` (`createLabReport.ts:26‚Äì36`) which checks the upper bound (`20 MB`), so the happy path is safe. But:

- Negative values are not rejected (`-1` passes `<= 20MB`).
- `NaN` is not rejected (`NaN > N` is `false`).
- The 20 MB cap is informational only to the partner (no "too large" response contract is documented to external labs).

The more concerning part is that `contentHash` is a client-supplied `"pending:${args.fileId}"` string ‚Äî a lab partner could pass any `fileId`, including one pointing at someone else's uploaded blob, because `_storage` IDs aren't scoped. But since Phase 4 adds real partner-token auth, that's a known stub concern, not a new one.

**Why it matters:**

- `fileSizeBytes` feeds telemetry and may feed future rate-limit decisions. Negative/NaN silently pollutes.
- External-partner surface = defense-in-depth is free here. One `if (!(args.fileSizeBytes > 0) || !Number.isFinite(args.fileSizeBytes))` throw is a one-liner.

**How to fix:**

Tighten `validateSizeAndMime` in `createLabReport.ts` (one site, two callers benefit) to also reject non-finite and non-positive sizes:

```ts
if (!Number.isFinite(args.fileSizeBytes) || args.fileSizeBytes <= 0) {
  throw new Error("invalid_file_size");
}
if (args.fileSizeBytes > MAX_FILE_SIZE_BYTES) {
  throw new Error("file_too_large");
}
```

Add one test for each. This change benefits `intakeUpload` and `simulateLabUpload` too.

#### M-2: Doctor query's N+1 is fine for a stub but should have a one-line complexity note

**File:** `convex/biomarker/portal/biomarkerReportsForPatient.ts:28‚Äì37`

The `for (const report of reports)` loop runs one index read per report. For a typical patient (1‚Äì5 reports historically) this is fine. For a long-history patient with 50+ reports it's 50+ round trips per doctor page-load. Not a blocker in 2.5, but:

No code change required, one comment would earn its keep.

## Recommendations

1. **Resolve C-1 before merge.** Pick option 1 (session-token, matches codebase) unless there's a specific reason to keep the `getUserIdentity` path. If option 2, get it in writing as a decision note that extends `2026-04-20-session-token-auth-in-2-5c-mutations.md` ‚Äî don't let the two auth styles drift silently.

2. **Hoist the prod-detection regex to one module.** `convex/lib/envGuards.ts` exporting `assertNotProd()`, `PROD_DEPLOYMENT_PATTERNS`, and `isProdDeployment(deployment)` ‚Äî then `portalGates.ts` imports `isProdDeployment` instead of rolling its own `PROD_PATTERN`. This kills I-1 permanently and sets the right pattern for every future env-guard surface.

3. **Add a wire-contract snapshot test for `biomarkerReportsForPatient`.** `vitest` snapshot of the query's return shape keyed on a seeded fixture. When someone adds a field to `biomarker_reports`, the test fails and forces a conversation about whether it should be doctor-visible. Cheap guardrail for a schema that will evolve.

4. **Smoke test path for `simulateLabUpload`.** The commit message notes "No dashboard smoke yet (requires convex auth)" ‚Äî that's the C-1 issue. Once C-1 is resolved, run the dashboard smoke the plan calls for (Task 29 Step 2) and check in a screenshot or a short note. The plan explicitly lists this as a step; skipping it silently is scope drift.

5. **Update `checkpoint.md` now, not at end-of-phase** (see I-4). Every commit log + test count + deviation deserves a live line in the checkpoint. This is a project rule, not a style preference.

6. **Process: 2.5A review's tightening of `PROD_DEPLOYMENT_PATTERNS` was a hard-won decision. Wave 5 reverting to `/prod/i` suggests the previous review's findings aren't being scanned before adding new env-guard code.** Consider adding a line to the plan template: "Before adding a `process.env` read, search existing reviews for the env-name ‚Äî prior review findings may apply."
